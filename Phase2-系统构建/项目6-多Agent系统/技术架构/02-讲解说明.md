## 一、开场：用"单人 vs 团队"讲多 Agent

老板，**这个项目学员容易陷入"过度设计"——以为 Agent 越多越好**。先纠偏：

> "多 Agent ≠ Agent 越多越好。**一个 Agent 解决不了的问题，拆给 4-5 个就够。超过 10 个 Agent 的系统，90% 是设计有问题**。"

**真实业务比喻**：

> "你去一家公司，CEO 自己写代码、自己做销售、自己做客服——**累死他，效率也低**。
> 
> 正确做法是：CEO 管 HR，HR 管研发和销售，研发写代码，销售接单，客服售后——**各司其职**。
> 
> 多 Agent 系统就是一家公司——**每个 Agent 是一个员工**。"

**强调工业价值**：

> "你去 Boss 直聘搜 'Agent 工程师'，**'多 Agent 协作经验' 是高频词**。大厂的客服系统、内容生成、代码助手，**全都在用多 Agent**。"

## 二、4 个核心概念，必须讲透

### 概念 1：多 Agent ≠ 多 Prompt

**学员最大误区**：

```python
# ❌ 学员的第一反应
def super_agent(query):
    if "调研" in query:
        return research_prompt(query)
    elif "分析" in query:
        return analysis_prompt(query)
    elif "写作" in query:
        return writing_prompt(query)
```

**这不叫多 Agent**，这叫"if-else 路由"。

**真正的多 Agent**：

> 每个 Agent 有**自己的目标、自己的工具、自己的记忆、自己的决策能力**。**Agent 间通过消息/状态通信**。

**对比**：

| 维度 | 多 Prompt | 多 Agent |
|------|-----------|----------|
| 状态 | 共享 | 独立/可共享 |
| 工具 | 共享 | 各自专属 |
| 决策 | 中心化 | 分布式 |
| 通信 | 函数调用 | 消息/状态 |
| 可观测 | 难 | 易 |

### 概念 2：角色设计 = Prompt 工程的高级版

**学员误区**：以为"角色"就是"提示词里加个'你是研究员'"。

**真相**：

> 好的角色设计 = **身份 + 目标 + 故事 + 工具** 四件套。

**反例**：

```python
# ❌ 烂角色
RESEARCHER_PROMPT = "你是研究员，负责搜索资料。"
```

**正例**：

```python
# ✅ 好角色
RESEARCHER_PROMPT = """你是王博士，资深市场研究员，专注科技行业 15 年。

[身份]
- 北京大学 MBA，曾任 Gartner 高级分析师
- 发表过 30+ 行业白皮书
- 习惯用数据和案例说话

[目标]
基于用户给定的主题，搜索最新（2024-2026）资料，整理 10 个关键洞察。

[工作方式]
1. 使用 search_web 工具搜索（中英文都要）
2. 使用 fetch_url 获取详细内容
3. 不编造数据，不确定的明确标注
4. 每个洞察附上来源链接

[输出格式]
JSON 格式，包含 key_points 数组。
"""
```

**讲师金句**：

> "**角色设计 = 给 AI 写'人物小传'**。**故事越具体，AI 越像那个角色**。"

### 概念 3：编排模式（3 种核心）

**讲师画图**（白板必备）：

```
1. 串行 (Sequential)
   A → B → C → D
   用于: 有明确依赖

2. 并行 (Parallel)
   ┌─ B ─┐
   A ──┼─ C ──→ D
   └─ E ─┘
   用于: 互相独立

3. 层级 (Hierarchical)
   Manager
   ├─ A1 → A2
   ├─ B1 → B2 → B3
   └─ C1
   用于: 复杂任务
```

**本项目用混合模式**：

```
研究员 → 分析师 → [写作者 ∥ 审校员] → 主编
         串行         并行         串行
```

**学员作业**：用 CrewAI 串行模式跑通 4 Agent，加深理解。

### 概念 4：状态管理（关键中的关键）

**学员最容易晕的部分**——**Agent 间怎么传数据**？

**答案：共享 State**。

```python
# 共享 State
class ReportState(TypedDict):
    research_data: dict   # 研究员写入
    analysis: dict        # 分析师写入
    draft: str            # 写作者写入
    review: dict          # 审校员写入
    final_report: str     # 主编写入
```

**流程**：

```
研究员 → 写 research_data → State 更新
                                    ↓
分析师 ← 读 research_data → 写 analysis → State 更新
                                                          ↓
写作者 ← 读 research_data + analysis → 写 draft → State 更新
```

**讲师金句**：

> "**State = Agent 间的'共享文档'**。**每个 Agent 读自己需要的，写自己负责的**。"

## 三、其他概念，根据学员水平选讲

### 概念 5：Human-in-the-Loop

**为什么需要**：

> AI 写报告可能跑偏、可能编造。**关键节点让人审核** = 质量兜底。

**实现**（LangGraph）：

```python
from langgraph.types import interrupt

async def human_review_node(state):
    # 触发中断，前端显示给用户
    user_input = interrupt({
        "draft": state["draft"],
        "ai_review": state["review"]
    })
    return {"user_decision": user_input}
```

**适用场景**：

- 报告生成（审校后）
- 客服回复（发送前）
- 金融决策（执行前）
- **任何"错了代价大"的场景**

### 概念 6：失败处理

**3 道防线**：

| 防线 | 机制 |
|------|------|
| 单步重试 | 失败 3 次就降级 |
| 状态恢复 | Checkpoint 持久化 |
| 全局超时 | 5 分钟硬超时 |

**讲师金句**：

> "**多 Agent 系统 = 多个失败点**。**每一步都可能挂**，必须有兜底。"

## 四、代码讲解：5 个 Demo

### Demo 1：CrewAI Hello World（20 分钟）

**为什么先讲 CrewAI**：

> CrewAI 上手最快。**让学员先建立多 Agent 的直觉**，再讲 LangGraph。

**核心代码**（5 行）：

```python
from crewai import Agent, Task, Crew

researcher = Agent(
    role="研究员",
    goal="搜索资料",
    backstory="你是资深研究员...",
    tools=[search_tool]
)

writer = Agent(
    role="写作者",
    goal="写文章",
    backstory="你是专业作家..."
)

task1 = Task(description="研究 AI 医疗", agent=researcher)
task2 = Task(description="写报告", agent=writer)

crew = Crew(agents=[researcher, writer], tasks=[task1, task2])
result = crew.kickoff()
```

**讲师要点**：

- `role` + `goal` + `backstory` = 角色 3 件套
- `tools` = 工具列表
- `tasks` 数组顺序 = 执行顺序
- **简单到学员第一眼就懂**

### Demo 2：LangGraph 4 Agent 串行（40 分钟）

**完整代码**（架构书里有详细版）：

```python
from langgraph.graph import StateGraph, START, END

class ReportState(TypedDict):
    topic: str
    research: str
    analysis: str
    draft: str
    review: str
    final: str

async def researcher(state):
    prompt = f"研究: {state['topic']}\n输出关键信息"
    return {"research": (await llm.ainvoke(prompt)).content}

# ... analyst, writer, reviewer, editor 类似

workflow = StateGraph(ReportState)
workflow.add_node("researcher", researcher)
workflow.add_node("analyst", analyst)
workflow.add_node("writer", writer)
workflow.add_node("reviewer", reviewer)
workflow.add_node("editor", editor)

workflow.add_edge(START, "researcher")
workflow.add_edge("researcher", "analyst")
workflow.add_edge("analyst", "writer")
workflow.add_edge("writer", "reviewer")
workflow.add_edge("reviewer", "editor")
workflow.add_edge("editor", END)

app = workflow.compile()
result = await app.ainvoke({"topic": "AI 医疗"})
print(result["final"])
```

**让学员亲手跑**：

- 输入 "AI 在医疗领域的应用"
- 看每个 Agent 的输出
- **重点观察 State 是怎么传递的**

### Demo 3：条件分支（30 分钟）

**加质量评分，不合格重写**：

```python
async def reviewer(state):
    prompt = f"""
审校以下报告:
{state['draft']}

输出 JSON:
{{
    "score": 8,
    "issues": ["..."],
    "feedback": "..."
}}
"""
    review = json.loads(extract_json((await llm.ainvoke(prompt)).content))
    return {"review": review, "score": review["score"]}

def should_rewrite(state):
    return "writer" if state["score"] < 7 else "editor"

workflow.add_conditional_edges(
    "reviewer",
    should_rewrite,
    {"writer": "writer", "editor": "editor"}
)
```

**演示**：
- 故意输入简单主题 → 报告 1 次通过
- 输入复杂主题 → 报告重写 2 次
- **学员看到自动循环的过程**

### Demo 4：人工审核（30 分钟）

```python
from langgraph.types import interrupt, Command

async def human_gate(state):
    """人工审核节点"""
    # 暂停，等待用户决策
    user_input = interrupt({
        "type": "review",
        "draft": state["draft"],
        "ai_score": state["score"]
    })
    
    if user_input["action"] == "approve":
        return {"score": 10, "user_approved": True}
    else:
        return {
            "review": {
                "feedback": user_input["feedback"],
                "score": 5
            },
            "user_approved": False
        }

# 前端代码（Streamlit）
import streamlit as st

if "graph_state" in st.session_state:
    state = st.session_state.graph_state
    
    if state.get("__interrupt__"):
        st.markdown("## AI 生成的报告")
        st.markdown(state["draft"])
        
        col1, col2 = st.columns(2)
        with col1:
            if st.button("✅ 通过"):
                graph.invoke(Command(resume={"action": "approve"}))
        with col2:
            feedback = st.text_input("修改意见")
            if st.button("❌ 打回"):
                graph.invoke(Command(resume={"action": "reject", "feedback": feedback}))
```

**演示**：
- Agent 写完报告 → 暂停
- 学员扮演用户，决策通过/打回
- 学员看到 Agent 根据反馈修改

### Demo 5：并行 Agent（挑战版，30 分钟）

**研究员 + 数据分析师同时跑**：

```python
async def parallel_research(state):
    """并行研究（多角度）"""
    tasks = [
        llm.ainvoke(f"研究 {state['topic']} 的市场角度"),
        llm.ainvoke(f"研究 {state['topic']} 的技术角度"),
        llm.ainvoke(f"研究 {state['topic']} 的用户角度")
    ]
    results = await asyncio.gather(*tasks)
    return {"research_angles": [r.content for r in results]}

# 加到 graph
workflow.add_node("parallel_research", parallel_research)
workflow.add_edge("parallel_research", "analyst")
```

**讲师要点**：

- `asyncio.gather` = 并行执行
- 速度提升 3 倍
- 成本提升 3 倍（**trade-off**）

## 六、必讲的 5 个踩坑点

### 坑 1：Agent 间状态污染

**现象**：研究员的状态传给了分析师，但分析师不需要。

**对策**：**State 设计要最小化**——每个 Agent 只读自己需要的。

### 坑 2：LLM 串行调用太慢

**现象**：4 Agent × 30s = 2 分钟。

**对策**：
- 能并行的就并行（`asyncio.gather`）
- 不需要深度的步骤用 `gpt-4o-mini`
- **报告生成场景可以节省 50% 时间**

### 坑 3：审校 Agent 太挑剔

**现象**：审校永远打 5 分，永远重写。

**对策**：
- 调低 Prompt 要求
- 多次重试后强制通过
- **用"达标而非完美"的心态**

### 坑 4：循环调用烧光 Token

**现象**：条件分支让 Agent 跑了 10 次。

**对策**：
- 设置最大迭代次数（`max_iteration=3`）
- 检测循环（同样的输入 → 同样的输出）
- **Token 成本 = Linear(循环次数)**

### 坑 5：Human-in-the-Loop 卡住

**现象**：用户离开，Agent 永远暂停。

**对策**：
- 默认 5 分钟自动通过/拒绝
- 提供 Web 通知
- **生产环境：异步审核 + Email 提醒**

## 七、Q&A 预案

**Q1：多 Agent 一定比单 Agent 好吗？**
A：**不一定**。简单任务单 Agent 更快。多 Agent 适合**复杂、有依赖、需要协作**的任务。

**Q2：4 个 Agent 够吗？要不要 10 个？**
A：**4-5 个是甜区**。超过 10 个，调试困难，token 成本飙升。

**Q3：怎么决定串行还是并行？**
A：看依赖。**A 的输出是 B 的输入 → 串行**。**A、B 互相独立 → 并行**。

**Q4：多 Agent 的状态怎么设计？**
A：**最小化 + 单一职责**。每个 Agent 只关心自己负责的字段。

**Q5：Human-in-the-Loop 必须吗？**
A：不一定。**取决于风险等级**。生成报告可加，聊天机器人不必加。

**Q6：怎么评估多 Agent 系统？**
A：4 个维度：①任务完成率 ②质量评分 ③响应时间 ④成本。

**Q7：LangGraph 和 CrewAI 怎么选？**
A：**学习用 CrewAI，生产用 LangGraph**。LangGraph 更灵活，CrewAI 更易上手。

**Q8：多 Agent 能不能跨服务？**
A：能。**Celery + Redis 做任务队列**，但**复杂度上升 10 倍**。**项目 6 建议单机跑通**。
