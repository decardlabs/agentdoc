## 二、3 个核心概念，必须讲透

### 概念 1：记忆 = 3 层 + 2 维

**3 层**（最经典的分层）：

```
┌────────────────────────────┐
│   工作记忆 (Working)        │ ← 当前对话窗口 (LLM Context)
│   "最近 5 轮对话"           │
└────────────────────────────┘
              ↓ 摘要/提取
┌────────────────────────────┐
│   短期记忆 (Short-term)     │ ← 原始对话历史 (PostgreSQL)
│   "今天所有对话"            │
└────────────────────────────┘
              ↓ 摘要/提取
┌────────────────────────────┐
│   长期记忆 (Long-term)      │ ← 压缩事实 (向量库)
│   "用户的画像、偏好、历史事件" │
└────────────────────────────┘
```

**2 维**（容易被忽略）：

| 维度 | 含义 | 实现 |
|------|------|------|
| **时间维** | 多久前发生 | created_at + 衰减权重 |
| **相关维** | 与当前话题有多相关 | Embedding 相似度 |

**讲师金句**：

> "好的记忆系统 = **保留最近的 + 保留相关的 + 丢弃其他**。人和机器都是这样。"

### 概念 2：状态管理 = Agent 的"工作流引擎"

**为什么单独讲**：

项目 4 引入了**状态机**概念（LangGraph），这是从"玩具"到"产品"的分水岭。

**对比**：

| 方式 | 状态管理 | 适用场景 |
|------|----------|----------|
| 普通 LLM 调用 | 无 | 单轮对话 |
| LangChain Agent | 简单 | 多轮但线性 |
| **LangGraph** | **状态机** | **复杂流程、可视化、可恢复** |

**画在白板上**：

```
START → recall_memories → generate → save_memories → END
         ↑                  ↓
         │           check_quality?
         └──── 不合格 ←──┘
```

**讲师要演示** LangGraph Studio（可视化工具）：

> "学员看到自己的 Agent 长什么样，是非常有成就感的一刻。"

### 概念 3：事实提取 = 「把对话变成结构化数据」

**学员误区**：以为"记忆 = 存对话历史"。

**真相**：

> 存对话历史是"录像"，事实提取是"做笔记"。

**反例对比**：

| 方式 | 存储 | 检索效率 |
|------|------|----------|
| **录像**（存原始对话） | "小明说：'我叫小明，在深圳'。AI 说：'你好...'。小明说：'我是产品经理'..." | 慢，要先 Embedding 整段 |
| **做笔记**（存事实） | "姓名=小明，城市=深圳，职业=产品经理" | **快，直接读** |

**演示**（必做）：

```
对话: "我叫小明，在深圳做产品经理，3 年经验，喜欢简洁的回答。"

提取后:
[
  {"type": "personal_info", "content": "用户叫小明"},
  {"type": "personal_info", "content": "用户在深圳"},
  {"type": "personal_info", "content": "用户是产品经理，3 年经验"},
  {"type": "preference", "content": "用户喜欢简洁回答"}
]
```

**讲师金句**：

> "**好的记忆不是'什么都存'，而是'只存重要的'**。事实提取就是 AI 的'记笔记'能力。"

## 三、其他概念，根据学员水平选讲

### 概念 4：检查点（Checkpoint）

**为什么需要**：

Agent 出错、网络中断、用户刷新页面——都需要**恢复到之前的状态**。

**实现**：

```python
# LangGraph 自带 Checkpoint
app = workflow.compile(checkpointer=PostgresSaver(...))

# 调用时指定 thread_id
config = {"configurable": {"thread_id": "session_123"}}
result = app.invoke({"messages": [...]}, config=config)

# 下次再用同一个 thread_id，自动恢复上下文
```

### 概念 5：跨用户隔离

**生产环境必做**——否则会出"用户 A 看到用户 B 的信息"的事故。

**3 道防线**：

```python
# 1. PostgreSQL 查询强制带 user_id
query = "SELECT * FROM conversations WHERE user_id = %s"
db.execute(query, (user_id,))

# 2. 向量库检索加 where 过滤
collection.query(
    query_texts=[query],
    where={"user_id": user_id}  # 必填
)

# 3. API 层加权限校验
@router.get("/memory/{user_id}")
async def get_memory(user_id: str, current_user = Depends(get_current_user)):
    if current_user.id != user_id and not current_user.is_admin:
        raise HTTPException(403, "无权访问")
```

**讲师金句**：

> "**跨用户数据泄漏 = 公司倒闭**。这个不是技术问题，是法律问题。"

### Demo 2：LangGraph + PostgresSaver（30 分钟）

**讲师带学员一步步搭**：

```python
# 第 1 步：定义状态
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]
    user_id: str

# 第 2 步：定义节点
def chat_node(state: AgentState):
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

# 第 3 步：构建图
workflow = StateGraph(AgentState)
workflow.add_node("chat", chat_node)
workflow.add_edge(START, "chat")
workflow.add_edge("chat", END)

# 第 4 步：加 Checkpoint
checkpointer = PostgresSaver.from_conn_string(DB_URL)
app = workflow.compile(checkpointer=checkpointer)
```

**重点讲**：
- `Annotated[list, add_messages]` —— 自动合并消息，不覆盖
- `thread_id` —— LangGraph 用它来区分会话
- Checkpoint 自动序列化所有 state

### Demo 3：长期记忆 + Chroma（40 分钟）

**这是项目 4 的重头戏**：

```python
async def recall_node(state: AgentState):
    """检索长期记忆"""
    query = state["messages"][-1].content
    
    # 1. 检索
    results = collection.query(
        query_texts=[query],
        n_results=3,
        where={"user_id": state["user_id"]}
    )
    
    # 2. 注入到 messages
    memories = results["documents"][0]
    if memories:
        context = "\n".join(f"- {m}" for m in memories)
        system_msg = SystemMessage(
            content=f"关于用户的已知信息:\n{context}\n\n请基于这些信息回答。"
        )
        return {"messages": [system_msg]}
    return {}
```

**学员常见问题**：
- "为什么用 where 过滤？"——必须按用户隔离
- "检索为空怎么办？"——优雅降级，正常回答
- "3 个事实够吗？"——**生产环境建议 5-10 个**

### Demo 4：事实提取（30 分钟）

```python
EXTRACT_FACT_PROMPT = """分析对话，提取值得记住的事实。

对话:
{conversation}

输出 JSON 数组，每项包含 type 和 content。
type 可选: personal_info / preference / plan / decision
"""

async def save_node(state: AgentState):
    """提取并保存事实"""
    # 1. 提取
    prompt = EXTRACT_FACT_PROMPT.format(...)
    response = await llm.ainvoke(prompt)
    facts = json.loads(response.content)
    
    # 2. 存储
    for fact in facts:
        embedding = await embed_model.aembed(fact["content"])
        collection.add(
            ids=[f"{state['user_id']}_{uuid4()}"],
            documents=[fact["content"]],
            embeddings=[embedding],
            metadatas=[{
                "user_id": state["user_id"],
                "type": fact["type"]
            }]
        )
    
    return {}
```

**讲师要点**：
- **异步执行**——不阻塞用户响应（用 `asyncio.create_task`）
- **去重**——相似事实不重复存
- **置信度**——提取的事实带 score，过滤低质量

### Demo 5：摘要压缩（30 分钟）

```python
async def compress_old_conversations(user_id: str):
    """压缩 30 天前的对话"""
    old = await db.get_conversations(user_id, days_ago=30)
    if len(old) < 20:
        return
    
    # 1. 摘要
    summary_prompt = f"将以下对话压缩成 200 字摘要:\n{old}"
    summary = await llm.ainvoke(summary_prompt)
    
    # 2. 存储摘要
    await db.save_summary(user_id, summary.content)
    
    # 3. 删除原始
    await db.delete_conversations(user_id, days_ago=30, keep=5)
```

**讲师金句**：

> "**没有摘要的 Agent = 永远涨价的账单**。原始对话越长，Token 越贵。摘要能把成本降 10 倍。"

## 六、必讲的 5 个踩坑点

### 坑 1：跨用户数据泄漏

**现象**：用户 A 检索到用户 B 的事实。

**原因**：
```python
# ❌ 错误
results = collection.query(query_texts=[query], n_results=3)  # 没 where 过滤
```

**对策**：永远加 `where={"user_id": user_id}`。

### 坑 2：Checkpoint 内存爆炸

**现象**：服务运行几天后内存爆炸。

**原因**：`PostgresSaver` 默认保留**所有历史**，1000 轮对话就有几万条消息。

**对策**：
```python
# 定期清理
async def cleanup_old_sessions():
    """清理 7 天前的会话"""
    await db.execute("DELETE FROM checkpoints WHERE created_at < NOW() - INTERVAL '7 days'")
```

### 坑 3：事实提取失败

**现象**：LLM 返回的不是合法 JSON。

**对策**：
```python
import json
import re

def parse_facts(response_text):
    # 三重保险
    try:
        return json.loads(response_text)
    except:
        pass
    
    # 尝试提取 JSON 块
    match = re.search(r'\[.*?\]', response_text, re.DOTALL)
    if match:
        try:
            return json.loads(match.group())
        except:
            pass
    
    # 失败返回空
    logger.warning(f"事实提取失败: {response_text}")
    return []
```

### 坑 4：摘要丢失关键信息

**现象**：摘要后用户问"我之前说的那个客户是谁？"，答不上来。

**对策**：
- 摘要 prompt 强调"保留人名、数字、决策"
- 重要事实**不要摘要**——直接存到长期记忆
- 关键事件用"标记"字段，**绕过摘要**

### 坑 5：记忆膨胀

**现象**：1 个用户存了 1000+ 条事实，检索变慢。

**对策**：
- 每月合并相似事实
- 删除置信度低的事实
- 限制单用户最大存储（1000 条）

## 七、Q&A 预案

**Q1：记忆会一直增长吗？怎么控制成本？**
A：会。生产环境需要：①定期摘要 ②合并相似 ③设置上限 ④按用户分层（免费用户少存，VIP 多存）。

**Q2：用户能删除记忆吗？**
A：**必须能**（GDPR/中国《个人信息保护法》）。提供"删除我的所有记忆"按钮。

**Q3：LangGraph 比 LangChain Memory 强在哪？**
A：LangChain Memory 是"包装"，LangGraph 是"框架"。LangGraph 能：①可视化 ②分支 ③人机协作 ④复杂状态。

**Q4：事实提取会让 LLM 调用翻倍吗？**
A：会。建议**异步执行**（不阻塞用户响应），用 gpt-4o-mini（成本可控）。

**Q5：能不能让用户看到 Agent 记住了什么？**
A：**强烈推荐**。提供 UI："以下是我记得的关于你的信息 [查看] [删除]"。增加透明度 + 信任。

**Q6：记忆被污染怎么办？（用户说"我 18 岁"又改口"我 25 岁"）**
A：存带时间戳，**新事实覆盖旧事实**，**保留修正历史**用于审计。

**Q7：跨设备的同一个用户怎么同步？**
A：用统一的 `user_id`，**所有设备用同一个 thread_id**。
