## 一、规划类问题

### Q1: 我没有 LLM 基础，能直接开始吗？

**A**: 建议先花 1-2 周补基础：

**必须具备**：
- Python 基础（异步编程、`*args`/`**kwargs`、装饰器）
- HTTP 请求（`requests` 库）
- JSON 数据处理
- 命令行使用
- Git 基础

**建议提前学习**：
- LLM 概念：Token、Prompt、Context Window、Temperature
- OpenAI API 基础：能完成一次 `chat.completions.create` 调用

**学习资源**：
- Python 基础：https://www.learnpython.org
- OpenAI 快速入门：https://platform.openai.com/docs/quickstart

---

### Q2: 没有 GPU，能跑开源模型吗？

**A**: 本培养计划主要使用 API（OpenAI/Claude/国产大模型），不需要本地 GPU。

**如果想用开源模型**：

| 模型大小 | 硬件要求 | 推荐方案 |
|----------|----------|----------|
| 7B | 16GB 内存 | M2 Mac / 入门级 GPU |
| 13B | 24GB 内存 | 24GB GPU（如 3090）|
| 70B | 48GB+ 显存 | A100 / 多卡 |
| 100B+ | 云 GPU | AutoDL / 恒源云 / RunPod |

**推荐路径**：
- 初学：用 API（便宜、方便）
- 进阶：本地跑 7B 模型（学习 vLLM、Ollama）
- 生产：用云服务（稳定、可扩展）

### Q3: 预算有限，怎么降低 API 成本？

**A**: 多种策略组合：

**模型选择**：
- 用便宜的模型（GPT-4o-mini、Claude Haiku、DeepSeek）
- 不同任务用不同模型（简单任务用便宜模型，复杂任务用贵模型）

**优化策略**：
- ✅ 加入缓存（相同问题直接返回）
- ✅ 控制 Context 长度（不要无脑塞历史对话）
- ✅ 用 RAG 减少 Prompt（不要把整个文档塞进去）
- ✅ 用国产模型（DeepSeek、Qwen 比 OpenAI 便宜 5-10 倍）

**成本估算**：
| 模型 | 1M Token 价格 | 1 万次问答 |
|------|---------------|-----------|
| GPT-4o-mini | $0.15 | ~$1-3 |
| GPT-4o | $5 | ~$30-100 |
| DeepSeek | ¥1 | ~¥10-30 |
| Qwen-Turbo | ¥0.3 | ~¥3-10 |

**建议**：开发阶段用便宜模型（GPT-4o-mini），生产环境再切到更强的模型。

## 二、技术类问题

### Q9: Function Calling 和 Tool Use 有什么区别？

**A**: 本质上是一回事，不同框架的叫法不同：

| 框架 | 叫法 |
|------|------|
| OpenAI | Function Calling / Tools |
| Anthropic | Tool Use |
| LangChain | Tools |
| LlamaIndex | Function Tools |

**核心机制**：
- LLM 识别需要调用的工具
- 输出工具名 + 参数（JSON 格式）
- 开发者执行工具
- 把结果反馈给 LLM
- LLM 生成最终回复

---

### Q10: LangChain 和 LlamaIndex 选哪个？

**A**: 看场景：

| 场景 | 推荐 |
|------|------|
| 复杂 Agent（多工具、多步骤）| LangChain |
| RAG 为主 | LlamaIndex |
| 都需要 | LangChain + LlamaIndex |
| 简单 LLM 应用 | 直接用 OpenAI SDK |

**学习建议**：
- 先学 LangChain（更通用）
- 再学 LlamaIndex（针对 RAG 优化）
- 深入后看 LangGraph（更底层的 Agent 框架）

---

### Q11: RAG 和 Fine-tuning 怎么选？

**A**: 看需求：

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新 | ✅ 实时 | ❌ 需要重新训练 |
| 成本 | 🟢 低 | 🔴 高 |
| 实现难度 | 🟢 简单 | 🔴 复杂 |
| 适合场景 | 文档问答、知识库 | 风格学习、特定任务 |

**混合使用**：
- 知识用 RAG（实时更新）
- 风格用 Fine-tuning（特定语气）
- 工具调用用 Fine-tuning（行为模式）

---

### Q12: Agent 一直循环怎么办？

**A**: 这是常见问题，多种解决方案：

**短期方案**：
```python
# 1. 设置最大迭代次数
max_iterations = 5

# 2. 检测重复工具调用
if same_tool_called_3_times:
    break

# 3. 超时机制
if total_time > 60s:
    break
```

**长期方案**：
- 优化 Prompt，让 Agent 更明确
- 用更小的工具集合（避免选择困难）
- 用更智能的模型（GPT-4 > GPT-3.5）

---

### Q13: LLM 一直幻觉（编造内容）怎么办？

**A**: 多管齐下：

**Prompt 层面**：
- 明确说"不知道就说不知道"
- 要求引用来源
- 降低 temperature

**架构层面**：
- 用 RAG（基于真实文档）
- 加入验证步骤（让 LLM 自我检查）
- 用多 Agent（一个生成、一个审核）

**数据层面**：
- 提供准确、完整的知识库
- 标注信息的可信度

---

### Q14: Token 消耗太大怎么优化？

**A**: 多种策略：

**Prompt 优化**：
- 删除冗余信息
- 用结构化格式
- 压缩历史对话

**架构优化**：
- 用 RAG 减少全量输入
- 用更便宜的模型
- 缓存重复请求

**工具**：
```python
# Token 计数
import tiktoken
encoding = tiktoken.encoding_for_model("gpt-4o-mini")
tokens = len(encoding.encode(text))
```

---

### Q15: 多 Agent 系统的成本怎么控制？

**A**: 多 Agent 系统 Token 消耗大是正常的：

**成本估算**：
- 单 Agent：~1 次 LLM 调用
- 4 Agent 顺序执行：~4-8 次 LLM 调用
- 4 Agent 并行执行：~4-8 次 LLM 调用（但快）

**优化策略**：
- 用便宜的模型（GPT-4o-mini）
- 限制每个 Agent 的输出长度
- 关键 Agent 用强模型，辅助 Agent 用弱模型
- 加入早停机制（质量达标就停止）

## 三、部署类问题

### Q16: Docker 镜像太大（>2GB）怎么办？

**A**: 多阶段构建 + 优化基础镜像：

**优化技巧**：
```dockerfile
# ❌ 不好
FROM python:3.11    # 900MB

# ✅ 好
FROM python:3.11-slim  # 150MB
```

**其他优化**：
- 清理 apt 缓存：`rm -rf /var/lib/apt/lists/*`
- 用 `--no-cache-dir` 安装 pip
- 用 `.dockerignore` 排除不必要文件
- 多阶段构建（构建用大镜像，运行时用小镜像）

**目标**：最终镜像 < 500MB

---

### Q17: K8s 部署太复杂，能用 Docker Compose 替代吗？

**A**: 视场景而定：

| 场景 | 推荐 |
|------|------|
| 个人项目 | Docker Compose |
| 小团队（< 10 人）| Docker Compose / Swarm |
| 中型项目 | K3s（轻量 K8s）|
| 大型项目 | 完整 K8s |

**Docker Compose 限制**：
- 单机部署
- 没有自动伸缩
- 没有自愈能力
- 没有滚动更新

**K8s 优势**：
- 自动伸缩
- 自愈（Pod 挂了自动重启）
- 滚动更新
- 服务发现

**建议**：
- 本培养计划基础版用 Docker Compose
- 挑战版用 K8s（学习 K8s 是加分项）

## 四、学习方法类问题

### Q19: 遇到 Bug 怎么高效排查？

**A**: 系统化排查方法：

**1. 复现 Bug**
- 找到最小复现条件
- 记录输入、输出、错误信息

**2. 假设验证**
- 提出假设："可能是 X 导致的"
- 验证假设：打印日志、断点、单步调试

**3. 隔离问题**
- 排除法：禁用某些功能
- 二分法：定位到具体模块

**4. 搜索解决方案**
- 错误信息 + 框架名
- GitHub Issues
- Stack Overflow
- 官方文档

**5. 提问**
- 准备清晰的提问模板：
  - 目标：要做什么
  - 现状：实际发生了什么
  - 尝试：已经尝试了什么
  - 代码：相关代码片段
  - 环境：Python 版本、依赖版本

### Q20: 怎么高效读 LLM 相关论文？

**A**: 三遍阅读法：

**第一遍：鸟瞰（30 分钟）**
- 读标题、摘要、结论
- 看图表
- 决定要不要深入读

**第二遍：理解（2-3 小时）**
- 读引言、方法、实验
- 跳过数学细节
- 记录关键 idea

**第三遍：复现（1-2 天）**
- 自己实现核心算法
- 跑实验
- 写复现报告

**推荐工具**：
- [arXiv-sanity](http://www.arxiv-sanity.com/)
- [Connected Papers](https://www.connectedpapers.com/)
- [Papers with Code](https://paperswithcode.com/)
