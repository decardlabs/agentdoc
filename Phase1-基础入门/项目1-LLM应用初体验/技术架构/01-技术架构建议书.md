## 2. 业务诉求与技术挑战

### 2.1 业务诉求
用户上传一份 PDF / Markdown / TXT 文档，应用返回该文档的智能问答能力。典型场景：

- 论文速读：「这篇论文的实验设置是什么？」
- 合同审查：「合同里关于违约的条款怎么写的？」
- 资料检索：「我们产品手册第 3 章讲了什么？」

### 2.2 核心挑战
| 挑战 | 表现 | 影响 |
|------|------|------|
| **上下文窗口有限** | GPT-4o 128K 上下文，Claude 200K，但成本随 Token 线性增长 | 不能把整本书丢给模型 |
| **长文档定位** | 用户的提问可能只对应文档的某一段 | 直接塞全文，召回率低、成本高 |
| **Token 成本** | GPT-4o 输入 $2.5/M Token，一本 500 页 PDF 约 25 万 Token ≈ ¥4/次 | 无脑塞全文，很快破产 |
| **答案准确性** | 模型可能"幻觉"，编造文档里没有的内容 | 失去用户信任 |

> 💡 **架构设计的核心矛盾**：用户希望"读完整本书"，但模型和钱包都不允许。

## 3. 整体架构

### 3.1 系统架构图

```
┌──────────────────────────────────────────────────────────┐
│                       用户浏览器                          │
│              (Streamlit Web UI / 终端 CLI)                │
└────────────────────────┬─────────────────────────────────┘
                         │ HTTP (上传文件 / 提问)
                         ▼
┌──────────────────────────────────────────────────────────┐
│                    应用服务层 (Python)                     │
│  ┌─────────────────┐         ┌──────────────────────┐    │
│  │   文档解析模块   │         │   问答服务 (FastAPI)  │    │
│  │  - PDF 切分      │         │  - 接收问题           │    │
│  │  - 文本清洗      │  ───►   │  - 构造 Prompt        │    │
│  │  - Chunking     │  索引   │  - 调用 LLM           │    │
│  └─────────────────┘         │  - 流式返回            │    │
│                               └──────────┬───────────┘    │
└──────────────────────────────────────────┼────────────────┘
                                          │ HTTPS (API)
                                          ▼
                              ┌──────────────────────┐
                              │   LLM Provider       │
                              │  (OpenAI / Claude)   │
                              └──────────────────────┘
```

### 3.2 数据流时序

**用户上传文档时**：

```
用户上传 PDF
  → 解析 (PyPDF2 / pdfplumber) 得到纯文本
  → 切分 (按 1000 字/段, 200 字重叠) 得到 Chunks
  → 存入内存字典 (doc_id -> List[Chunk])
  → 返回上传成功
```

**用户提问时**：

```
用户输入问题
  → 从内存取出该文档的全部 Chunks
  → 构造 Prompt: [系统提示] + [问题] + [全部 Chunks]
  → 调用 LLM (stream=True)
  → 流式返回答案
```

> ⚠️ **这是项目 1 的简化版本**（基础版）。完整 RAG（向量化 + 检索）放到项目 3。

### 3.3 核心模块职责

| 模块 | 职责 | 关键输入 | 关键输出 |
|------|------|----------|----------|
| **文档解析** | PDF/MD/TXT → 纯文本 | 文件二进制流 | 字符串 |
| **文本切分** | 纯文本 → Chunk 列表 | 长文本 | `List[str]` |
| **Prompt 构造器** | Context + Question → 最终 Prompt | 文档 Chunks、用户问题 | 完整 Prompt |
| **LLM 客户端** | 调用 OpenAI/Claude API | Prompt、流式开关 | 文本流 |
| **Web UI** | 文件上传、问题输入、流式展示 | 用户操作 | HTML |

## 4. 技术选型

### 4.1 选型决策表

| 层级 | 候选方案 | 选哪个 | 理由 |
|------|----------|--------|------|
| **LLM Provider** | OpenAI / Claude / 通义千问 | OpenAI（首选） | 文档最全、生态最成熟、gpt-4o-mini 性价比高 |
| **Web 框架** | Streamlit / Gradio / FastAPI + 前端 | **Streamlit**（基础版）| 5 行代码出 UI、专注业务逻辑 |
| **PDF 解析** | PyPDF2 / pdfplumber / Unstructured | pdfplumber | 中文支持好、能识别表格 |
| **文本切分** | LangChain TextSplitter / 手写 | LangChain `RecursiveCharacterTextSplitter` | 默认参数就够用、后续项目复用 |
| **配置管理** | 环境变量 / .env / pydantic-settings | `.env` + `python-dotenv` | 简单、便于切换 Key |

### 4.3 成本估算

假设用户平均提问 10 次/文档，每文档 50K Token：

| 模型 | 输入价格 | 单次提问成本 | 10 次成本 |
|------|----------|--------------|-----------|
| gpt-4o-mini | $0.15/M | $0.0075 | **$0.075** |
| gpt-4o | $2.5/M | $0.125 | **$1.25** |
| claude-3.5-sonnet | $3/M | $0.15 | **$1.5** |

> 基础版用 **gpt-4o-mini** 即可，10 次提问不到 1 块钱。挑战版用 gpt-4o 提升回答质量。

## 5. 关键技术点设计

### 5.1 Prompt 工程（核心中的核心）

**基础版 Prompt 模板**：

```python
SYSTEM_PROMPT = """你是一个文档问答助手。请严格基于以下文档内容回答问题。

规则：
1. 只使用文档中明确出现的信息
2. 文档里没有的内容，明确说"文档未提及"
3. 引用具体数据/条款时，标注所在的段落
4. 保持简洁，控制在 200 字以内

文档内容：
{doc_content}
"""

USER_PROMPT_TEMPLATE = "{question}"
```

**为什么这么设计**：
- 「规则 1-2」防幻觉
- 「规则 3」方便用户溯源
- 「规则 4」控制输出长度 = 控制成本

**进阶技巧（挑战版可学）**：

| 技巧 | 做法 | 收益 |
|------|------|------|
| **Few-shot** | 在 System Prompt 里加 1-2 个问答示例 | 提升回答格式一致性 |
| **Chain-of-Thought** | 加一句"请先分析问题涉及哪部分内容，再回答" | 提升复杂问题准确率 |
| **Citation 强制** | 加规则"必须用 [段落 X] 标注引用" | 便于溯源、增强可信度 |

### 5.2 文档切分策略

**为什么必须切分**：
- 500 页 PDF ≈ 25 万 Token，全塞 Prompt = 单次成本 ¥4
- 模型注意力衰减，超长 Context 后段内容易被忽略

**基础版切分**：

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,        # 每段 1000 字符
    chunk_overlap=200,      # 段间重叠 200 字符（防切断语义）
    separators=["\n\n", "\n", "。", "，", " "]  # 中文友好
)
chunks = splitter.split_text(doc_text)
```

**挑战版可优化**：

| 优化点 | 做法 | 适用场景 |
|--------|------|----------|
| **按章节切** | 解析 PDF 目录，按章节切 | 结构化文档（论文、书籍） |
| **按页切** | 每页一段 | 短文档、需要页码引用 |
| **语义切分** | 用 Embedding 模型检测语义断点 | 高质量需求，成本高 |

### 5.3 Token 控制（成本意识的第一课）

**问题**：Prompt 越长，成本越高。**怎么知道当前 Prompt 多长？**

```python
import tiktoken

def count_tokens(text: str, model: str = "gpt-4o-mini") -> int:
    encoding = tiktoken.encoding_for_model(model)
    return len(encoding.encode(text))

# 使用
prompt = f"{SYSTEM_PROMPT}\n\n{USER_PROMPT}"
print(f"Prompt 长度: {count_tokens(prompt)} tokens")
```

**成本控制策略**：

| 策略 | 做法 | 节省效果 |
|------|------|----------|
| **模型分级** | 简单问题用 gpt-4o-mini，复杂问题用 gpt-4o | 节省 50-80% |
| **输出限制** | Prompt 加 "请控制在 200 字以内" | 节省 30-50% |
| **历史压缩** | 长对话只保留最近 5 轮 + 早期对话摘要 | 节省 60%+ |
| **结果缓存** | 相同问题直接返回缓存 | 节省 90%+（适合 FAQ） |

### 5.4 错误处理与重试

**必须处理的 3 类错误**：

```python
import time
from openai import OpenAI, RateLimitError, APIError

client = OpenAI()

def call_llm_with_retry(prompt: str, max_retries: int = 3):
    for attempt in range(max_retries):
        try:
            return client.chat.completions.create(
                model="gpt-4o-mini",
                messages=[{"role": "user", "content": prompt}],
                stream=True
            )
        except RateLimitError:
            wait = 2 ** attempt  # 指数退避：1s, 2s, 4s
            print(f"限流，{wait}秒后重试")
            time.sleep(wait)
        except APIError as e:
            if attempt == max_retries - 1:
                raise
            time.sleep(1)
    raise Exception("重试失败")
```

**其他必须处理的场景**：
- 网络超时 → 重试 + 用户提示
- API Key 无效 → 友好报错，不暴露 Key
- 文件解析失败 → 提示「请上传有效的 PDF 文件」
- Prompt 超长 → 提示「文档过长，请尝试挑战版的分段处理」

## 7. 常见坑与对策

| 坑 | 表现 | 对策 |
|----|------|------|
| **API Key 泄漏** | 写死在代码里、提交到 GitHub | 用 `.env`，加进 `.gitignore` |
| **PDF 解析出乱码** | 扫描版 PDF 提取出空格/乱码 | 项目 1 不处理扫描版；项目 3 引入 OCR（paddleocr） |
| **Prompt 超长** | 文档太大，Prompt 超过模型 Context | 切分 + 摘要 + 分段处理（挑战版） |
| **响应太慢** | 用户等 10 秒才看到答案 | 启用 `stream=True` 流式输出 |
| **中文乱码** | 终端打印 `???` | 加 `print(..., flush=True)` + UTF-8 编码 |
| **幻觉严重** | 模型编造文档里没有的内容 | 强化 Prompt 规则 + 强制引用 |
