## 二、6 个核心概念，必须讲透

### 概念 1：Embedding = 「文本的指纹」

学员最大困惑：「向量是什么？文本怎么能变成数字？」

**比喻**：

> 把每个句子想象成地图上的一个点。
> - 意思相近的句子，离得近
> - 意思不同的句子，离得远

**演示**（必做）：

```python
from openai import OpenAI
client = OpenAI()

def embed(text):
    return client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    ).data[0].embedding

# 看 3 个句子的向量
v1 = embed("今天天气很好")
v2 = embed("阳光明媚")
v3 = embed("电脑蓝屏了")

print(f"v1 vs v2 相似度: {cosine(v1, v2):.3f}")  # 0.85
print(f"v1 vs v3 相似度: {cosine(v1, v3):.3f}")  # 0.12
```

让学员**亲眼看到**：意思相近的句子，向量也相近。

**讲师金句**：

> "Embedding 就像给每个句子算一个'语义 GPS 坐标'。**意思相近 = 坐标相近**。"

### 概念 2：相似度 ≠ 完全相同

**学员误区**：以为 RAG 就是「关键词搜索的升级版」。

**真相**：

| 检索方式 | 原理 | 局限 |
|----------|------|------|
| **关键词搜索** | 字符串匹配 | "退款" 找不到 "退货流程" |
| **向量检索** | 语义匹配 | "退款" 找得到 "退货流程" |

**演示**（必做）：

```
知识库里的文档: "用户购买商品后，如需退货，请联系客服..."
用户问题: "我买了东西不想要了怎么办？"

关键词搜索: 0 个匹配（没有"退货"两个连续字）
向量检索: 找到！因为语义相近
```

### 概念 3：Chunk = 检索的最小单位

**为什么不能整篇文档当一个 Chunk**：

```
整本《员工手册》(20 万字)
  ↓ Embedding
1 个向量 (1536 维)

问题: 用户问"年假怎么申请"
检索: 返回整本手册 (20 万字)
LLM: Prompt 装不下 → 报错 / 答错
```

**正确做法**：

```
整本《员工手册》
  ↓ 切分 (chunk_size=512)
400 个 Chunk
  ↓ Embedding
400 个向量

问题: 用户问"年假怎么申请"
检索: 返回最相关的 3 个 Chunk (~1500 字)
LLM: 答案精准 + 引用清晰
```

### 概念 4：检索 ≠ 答案

**学员误区**：以为"检索到了，答案就有了"。

**真相**：

```
检索到 (Recall) ≠ 答案对 (Accuracy)
```

**反例**：

```
问题: "公司有什么福利？"
检索到: 1 个 Chunk 说"公司提供五险一金"
LLM 答案: "公司有五险一金"

问题: "我刚入职有什么福利？"
检索到: 1 个 Chunk 说"五险一金、生日礼物、年度体检"
LLM 答案: "公司提供五险一金"  ← 漏了 2 项

原因: Chunk Size 太小，被切断了
```

**讲师要点**：检索质量决定了答案上限，LLM 只能逼近这个上限。

### 概念 5：Rerank = 二次精排

**问题**：向量检索的 Top-5 不一定都相关。

**比喻**：

> 向量检索是"粗筛"（快速但可能错），Rerank 是"精排"（慢但准）。

```
向量检索: 找 20 个候选 (快，但有错)
Rerank:  从 20 个里选 3 个最相关的 (慢，但准)
```

**数据**：

| 指标 | 无 Rerank | 有 Rerank |
|------|----------|-----------|
| Recall@3 | 60% | **85%** |
| 延迟 | 50ms | 200ms |

**讲师金句**：

> "Rerank 是 RAG 系统的'调味料'——加一点就香很多，但**不要加太多**（成本高）。生产环境**用 Rerank**。"

### 概念 6：评估 = 系统的体检表

**学员最忽略的环节**：写完就跑，不评估。

**真相**：

> 没有评估指标的 RAG 系统 = 没有仪表盘的汽车。
> 你不知道它跑得快不快、跑得对不对。

**必讲的 3 个指标**：

| 指标 | 含义 | 目标值 |
|------|------|--------|
| **Recall@K** | Top-K 中包含正确答案的比例 | ≥ 80% |
| **MRR** | 正确答案排名的倒数 | ≥ 0.7 |
| **P95 延迟** | 95% 请求的响应时间 | < 3s |

**讲师金句**：

> "RAG 工程师的面试 90% 会问评估——**答不出 Recall@K 的不要**。"

## 三、代码讲解：5 个 Demo，层层递进

### Demo 1：5 行代码跑通 RAG（15 分钟）

```python
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

# 1. 加载文档
documents = SimpleDirectoryReader("./data").load_data()

# 2. 索引（自动 Embedding + 存内存）
index = VectorStoreIndex.from_documents(documents)

# 3. 查询
query_engine = index.as_query_engine()
response = query_engine.query("公司有什么福利？")
print(response)
```

**让学员先跑起来**，建立信心。再逐步深入。

### Demo 2：自定义切分（20 分钟）

```python
from llama_index.core.node_parser import SentenceSplitter

# 自定义切分
text_splitter = SentenceSplitter(
    chunk_size=512,    # 每块 512 Token
    chunk_overlap=50
)

# 应用切分
from llama_index.core.ingestion import IngestionPipeline
pipeline = IngestionPipeline(
    transformations=[text_splitter, embed_model]
)
nodes = pipeline.run(documents=documents)
```

**重点讲**：为什么 `chunk_size=512`？太大太小都不行。

### Demo 3：加 Chroma 持久化（20 分钟）

```python
import chromadb
from llama_index.vector_stores.chroma import ChromaVectorStore

# 1. 创建 Chroma 客户端（持久化到本地）
chroma_client = chromadb.PersistentClient(path="./chroma_db")

# 2. 创建集合
chroma_collection = chroma_client.get_or_create_collection("my_rag")

# 3. 关联到 LlamaIndex
vector_store = ChromaVectorStore(chroma_collection=chroma_collection)

# 4. 构建索引（存入 Chroma）
storage_context = StorageContext.from_defaults(vector_store=vector_store)
index = VectorStoreIndex.from_documents(
    documents, storage_context=storage_context
)

# 5. 查询
query_engine = index.as_query_engine()
response = query_engine.query("年假怎么申请？")
```

**重点讲**：为什么需要持久化？——**重启不丢数据**。

### Demo 4：加元数据过滤（30 分钟）

**场景**：知识库有多个文档类别（人事、财务、技术），用户问"技术部"的问题只想检索技术文档。

```python
# 1. 加载时加元数据
documents = SimpleDirectoryReader(
    "./data",
    file_metadata=lambda filename: {"category": detect_category(filename)}
).load_data()

# 2. 查询时加过滤
from llama_index.core.vector_stores import MetadataFilter, MetadataFilters

filters = MetadataFilters(
    filters=[MetadataFilter(key="category", value="人事")]
)
query_engine = index.as_query_engine(filters=filters)
```

**重点讲**：元数据是 RAG 系统的"标签系统"——**没有元数据 = 没法精细化检索**。

### Demo 5：加 Rerank（挑战版，30 分钟）

```python
from llama_index.core.postprocessor import SentenceTransformerRerank

# 加 Rerank
rerank = SentenceTransformerRerank(
    model="BAAI/bge-reranker-large",
    top_n=3  # 最终返回前 3
)

query_engine = index.as_query_engine(
    similarity_top_k=10,  # 粗筛 10 个
    node_postprocessors=[rerank]  # 精排到 3 个
)
```

**重点讲**：粗筛 + 精排的两阶段检索。

## 四、必讲的 6 个踩坑点

### 坑 1：Chunk Size 选错

| 现象 | 原因 | 对策 |
|------|------|------|
| 答案缺上下文 | Chunk 太小 | 提到 512-1024 |
| 答案不精准 | Chunk 太大 | 降到 256-512 |
| Chunks 太多 | Chunk 太小 | 增大或合并 |

### 坑 2：Embedding 模型不一致

```python
# ❌ 索引用 A 模型
index = VectorStoreIndex.from_documents(
    documents,
    embed_model=OpenAIEmbedding(model="text-embedding-3-small")
)

# 查询用 B 模型
query_engine = index.as_query_engine(
    embed_model=OpenAIEmbedding(model="text-embedding-3-large")
)

# 结果: 检索全错（向量空间不一样）
```

**对策**：始终用同一个模型，封装成全局变量。

### 坑 3：PDF 解析失败

**现象**：扫描版 PDF 提取出空格 / 乱码。

**对策**：

| PDF 类型 | 工具 |
|----------|------|
| 文本型 PDF | `pypdf` / `pdfplumber` |
| 扫描版 PDF | `paddleocr` / `pymupdf` + OCR |
| 表格型 PDF | `pdfplumber`（自带表格识别） |

### 坑 4：检索不相关但 LLM 强行答

**现象**：检索到的 Chunk 不相关，LLM 编造答案。

**对策**：

```python
SYSTEM_PROMPT = """基于以下 Context 回答。

规则：
1. 只用 Context 里的信息
2. 如果 Context 不相关或为空，明确说"知识库中未找到相关信息"
3. 不要编造
"""
```

### 坑 5：不评估就上线

**现象**：学员写完直接交差。

**对策**：强制要求：
- 至少 30 个测试问题
- Recall@5 报告
- 至少 3 个失败案例分析

### 坑 6：忽略性能

**现象**：单个查询 10 秒才返回。

**原因**：
- 文档太多，检索慢
- LLM 生成慢（没开 stream）
- 没有缓存

**对策**：

| 优化 | 效果 |
|------|------|
| 启用 stream | 体验提升 10x |
| 加缓存 | 重复查询秒回 |
| 批量 embedding | 索引速度 5x |

## 六、Q&A 预案

**Q1：RAG 和 Fine-tuning 怎么选？**
A：
- **RAG**：知识频繁更新、需要引用溯源、成本低
- **Fine-tuning**：需要模型学会特定风格/格式/推理逻辑
- 现实：**90% 场景用 RAG** 就够了

**Q2：向量数据库能存多少数据？**
A：取决于实现。Chroma 单机能存百万级，Qdrant 千万级，Milvus 亿级。

**Q3：Embedding 模型需要 Fine-tuning 吗？**
A：90% 场景不需要。**领域术语特殊**（如医学、法律）才需要。

**Q4：怎么评估 RAG 系统的好坏？**
A：3 个维度：①检索质量（Recall@K）②答案质量（LLM-as-Judge）③性能（延迟）。**Ragas 工具**可以自动算。

**Q5：中文 RAG 有什么特殊点？**
A：
- Embedding 模型选 BGE（中文效果好）
- 切分时考虑中文标点（。、，）
- 如果混合中英文，用 BGE 多语言版

**Q6：项目 3 完，能找到工作吗？**
A：**能**。"RAG 工程师" 是 2024-2025 招聘需求最大的 AI 岗位之一。**这一个项目值得**。
