## 2. 业务诉求与技术挑战

### 2.1 业务诉求

构建一个**基于私有知识库**的问答系统，典型场景：

- 企业知识库：内部 Wiki、规章制度、产品手册
- 客服系统：FAQ + 工单历史 + 产品文档
- 学术研究：1000 篇论文 + 个人笔记
- 法律咨询：法律法规 + 判例 + 合同模板

**核心能力**：用户的提问能**精准定位到相关文档片段**，并基于片段生成答案。

### 2.2 核心挑战

| 挑战 | 表现 | 影响 |
|------|------|------|
| **语义匹配** | 用户问"怎么退款"，文档写"退货流程"——关键词不同 | 关键词检索召回率低 |
| **长文档检索** | 1000 页手册，提问只对应其中 1 段 | 整本塞给模型 = 慢+贵+错 |
| **多文档管理** | 10000 个文档，怎么高效找到相关那几个 | 暴力搜索不可行 |
| **答案溯源** | 用户问"这答案从哪来的" | 没有引用 = 用户不信任 |
| **新鲜度** | 文档每天更新，怎么同步 | 检索到过期内容 = 答错 |
| **幻觉防御** | 模型可能"编造"答案 | 比没答案更糟 |

> 💡 **RAG 的本质**：把"大海捞针"问题，变成"分块索引 + 语义检索 + LLM 合成"。

## 3. 整体架构

### 3.1 经典 RAG Pipeline

```
┌────────────────────────────────────────────────────────────┐
│                    离线索引阶段（一次或定期）                │
│                                                            │
│  文档集合 → 文档加载 → 文本切分 → Embedding → 向量数据库    │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│                    在线检索阶段（每次查询）                  │
│                                                            │
│  用户问题 → Query 改写 → Embedding → 向量检索 → Rerank     │
│                                          ↓                │
│                              Top-K 文档片段                 │
│                                          ↓                │
│                          Prompt 构造 → LLM → 答案           │
└────────────────────────────────────────────────────────────┘
```

### 3.2 数据流详解

#### 离线索引（Indexing）

```
原始文档 (PDF/MD/HTML)
  ↓ [1] 文档加载 (Loader)
纯文本 + 元数据（文件名、章节、页码）
  ↓ [2] 文本切分 (Splitter)
Chunks (List[Document])  // 每块 500-1000 字
  ↓ [3] Embedding 模型
Vectors (List[float[1536]])  // 每个 Chunk 一个向量
  ↓ [4] 存入向量数据库
VectorDB (持久化存储，支持相似度检索)
```

#### 在线检索（Retrieval）

```
用户问题
  ↓ [1] Query 预处理 (可选: Query 改写/扩展)
优化后的问题
  ↓ [2] Embedding
Query Vector
  ↓ [3] 向量检索 (ANN 搜索)
Top-K Candidates (默认 K=5-10)
  ↓ [4] Rerank (可选, 提升精度)
Top-N (N ≤ K, 通常 3-5)
  ↓ [5] Prompt 构造
[System: 基于以下 Context 回答...] + [Context: Chunks] + [User: 问题]
  ↓ [6] LLM 生成
最终答案 (含引用)
```

## 5. 关键技术点设计

### 5.1 文档切分（Chunking）

**为什么必须切分**：
- Embedding 模型有**输入长度限制**（OpenAI 是 8192 Token）
- 整篇文档变成 1 个向量 = 失去细粒度信息
- 检索时返回 1 个长文档 = Prompt 装不下

**基础版切分策略**：

```python
from llama_index.core.node_parser import SentenceSplitter

splitter = SentenceSplitter(
    chunk_size=512,        # 每块 512 Token (不是字符)
    chunk_overlap=50       # 重叠 50 Token，保持上下文
)
nodes = splitter.get_nodes_from_documents(documents)
```

**挑战版进阶**：

| 切分策略 | 适用场景 | 优劣 |
|----------|----------|------|
| **固定长度** | 通用 | 简单但可能切断语义 |
| **按句子** | 短文本 | 粒度细，检索准但 Chunks 多 |
| **按段落** | 结构化文档 | 自然语义，PDF 解析复杂 |
| **按章节** | 论文、书籍 | 保留逻辑结构，依赖目录 |
| **语义切分** | 高质量需求 | Embedding 聚类，成本高 |
| **滑动窗口** | 通用 + 召回 | 重叠保证不切断 |

**Chunk Size 选择经验值**：

| Chunk Size | 检索粒度 | 适用 |
|------------|----------|------|
| 128-256 | 细粒度 | FAQ、问答 |
| 512-1024 | 中粒度 | **推荐起点** |
| 1024-2048 | 粗粒度 | 长篇报告、综述 |

**Overlap 的作用**：

```
Chunk 1: "RAG 是一种检索增强生成技术。它结合了..."
Chunk 2: "结合了向量检索和大语言模型的优势..."
                   ↑
              overlap 区域（保留上下文）
```

避免「句子在边界被切断」。

### 5.2 Embedding 模型

**选 OpenAI text-embedding-3-small 的理由**：

| 模型 | 维度 | 性能 | 价格 | 推荐场景 |
|------|------|------|------|----------|
| **text-embedding-3-small** | 1536 | 优 | $0.02/M | 首选（性价比高） |
| text-embedding-3-large | 3072 | 优+ | $0.13/M | 高质量需求 |
| **BGE-large-zh-v1.5** | 1024 | 优 | 免费 | **中文场景、本地部署** |
| M3E-base | 768 | 中 | 免费 | 备选 |

**为什么维度重要**：

```
高维 (3072)  → 表达能力强，但存储 + 计算成本高
低维 (768)   → 速度快，但精度可能不够
中维 (1536)  → 平衡，OpenAI 默认
```

**关键概念**：

- **余弦相似度**（Cosine Similarity）：两个向量的夹角，越小越相似
- **点积**（Dot Product）：等价于余弦相似度（向量归一化后）
- **L2 距离**（欧氏距离）：值越小越相似

**Embedding 的局限**：

> Embedding 把文本压缩成固定向量，**会丢失信息**。两个语义相似但关键词完全不同的句子，可能相似度很低。

这就是为什么要加 **Rerank** 阶段。

### 5.3 向量检索（Vector Search）

**核心算法**：ANN（Approximate Nearest Neighbor）

```
暴力搜索: 遍历所有向量 → O(N)
ANN 搜索: 用 HNSW/IVF 索引 → O(log N)
```

**主流向量数据库对比**：

| 数据库 | 部署 | 性能 | 适用 |
|--------|------|------|------|
| **Chroma** | 嵌入式/服务端 | 中 | 开发原型、小规模 |
| **FAISS** | 嵌入式 | 高 | 离线处理、研究 |
| **Qdrant** | 服务端 | 高 | **生产推荐**（Rust 写） |
| Milvus | 分布式 | 极高 | 大规模（亿级） |
| Pinecone | 云服务 | 高 | 全托管，按量付费 |
| Weaviate | 服务端 | 中高 | 多模态支持好 |

**基础版用 Chroma 即可**。生产换 Qdrant。

### 5.4 Rerank（提升精度的关键）

**问题**：向量检索的 Top-5 不一定真的相关。

**解法**：用 Rerank 模型**重新打分**。

```
向量检索 Top-20 → Rerank 模型精排 → Top-3
```

**Rerank 模型**：

| 模型 | 性能 | 成本 | 推荐 |
|------|------|------|------|
| **Cohere Rerank-3** | 优 | $2/M Token | 首选 |
| **BGE Reranker v2** | 优 | 免费 | **中文场景、本地** |
| Jina Rerank | 良 | $0.018/query | 备选 |

**实际效果**：

```
无 Rerank:  Top-5 中 3 个相关 → 召回率 60%
有 Rerank:  Top-3 中 3 个相关 → 召回率 100%
```

### 5.5 Prompt 构造

**核心 Prompt 模板**：

```python
SYSTEM_PROMPT = """你是一个知识库问答助手。

## 规则：
1. 严格基于以下 Context 回答，不要编造
2. 引用 Context 时，标注来源 (例如: [文档 1, 段落 2])
3. Context 中没有的内容，明确说"知识库中未找到相关信息"
4. 答案简洁，控制在 300 字以内

## Context：
{context}
"""

USER_PROMPT = "{question}"
```

**Context 怎么填**：

```python
context = "\n\n".join([
    f"[{i+1}] (来源: {doc.metadata['source']})\n{doc.text}"
    for i, doc in enumerate(retrieved_docs)
])
```

**关键设计**：
- 给每段**编号 + 来源**，让 LLM 能引用
- 用**结构化格式**（`[1]`、`[2]`）便于溯源
- **不要塞过多 Context**，3-5 段就够

### 5.6 高级 RAG 技巧

#### Query 改写（Query Rewriting）

**问题**：用户问「怎么退款」，知识库里的文档写「退货流程」。

**解法**：让 LLM 把用户问题改写成多个变体。

```python
def rewrite_query(question: str) -> List[str]:
    prompt = f"""基于以下问题，生成 3 个不同表述用于检索：
    原问题：{question}
    
    变体（每行一个）："""
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}]
    )
    return [q.strip() for q in resp.choices[0].message.content.split("\n") if q.strip()]

# 使用
queries = [original_question] + rewrite_query(original_question)
all_results = []
for q in queries:
    all_results.extend(retriever.retrieve(q))

# 去重 + 重排序
```

#### HyDE（Hypothetical Document Embeddings）

让 LLM **先想象一个答案**，再用答案的 Embedding 去检索——比直接用问题检索更准。

#### Self-RAG

让 LLM 评估检索结果是否相关，无关则重新检索。

#### GraphRAG

把文档建成知识图谱，检索时同时考虑实体关系。

> **讲师要点**：基础版只讲**经典 RAG**。这些高级技巧列在挑战版，**让学员了解全貌，知道有这条路**。

## 6. 评估方法

**RAG 评估的 3 个维度**：

| 维度 | 指标 | 怎么测 |
|------|------|--------|
| **检索质量** | Recall@K, MRR, NDCG | 人工标注 50 个测试问题 |
| **答案质量** | 准确性、相关性、流畅性 | LLM-as-Judge 或人工评分 |
| **性能** | 检索延迟、端到端延迟 | 压测脚本 |

**推荐工具**：

- **Ragas**：RAG 评估框架（指标自动化）
- **TruLens**：评估 + 监控
- **人工标注**：最准，但贵

**最小评估集**：

```python
test_cases = [
    {
        "question": "怎么申请年假？",
        "expected_source": "员工手册.docx",
        "expected_keywords": ["年假", "申请", "HR"]
    },
    # ... 至少 30 个
]
```

**评估脚本（基础版）**：

```python
def evaluate(test_cases, retriever):
    hits = 0
    for case in test_cases:
        results = retriever.retrieve(case["question"])
        if case["expected_source"] in [r.metadata["source"] for r in results]:
            hits += 1
    recall_at_5 = hits / len(test_cases)
    return f"Recall@5: {recall_at_5:.2%}"
```

## 8. 常见坑与对策

| 坑 | 表现 | 对策 |
|----|------|------|
| **Chunk Size 太大** | 检索返回整页内容，Prompt 装不下 | 降到 256-512 Token |
| **Chunk Size 太小** | 检索片段不完整，答案缺上下文 | 提到 512-1024 |
| **文档加载失败** | PDF 扫描版提取出空格 | 加 OCR（paddleocr） |
| **Embedding 不一致** | 索引用 A 模型，查询用 B 模型 | **始终用同一个模型** |
| **不返回来源** | 答案没错但用户不信 | 强制 Prompt 加引用 |
| **检索不相关** | 关键词不同，召回率低 | Query 改写 + Rerank |
| **向量数据库太大** | 内存爆了 | 用 HNSW 索引 + 量化 |
| **多语言混合** | 中英混杂检索效果差 | 选支持多语言的 Embedding（BGE） |
