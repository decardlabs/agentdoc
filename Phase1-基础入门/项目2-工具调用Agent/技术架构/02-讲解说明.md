## 二、5 个核心概念，讲透 3 个就够

项目 2 信息量大，**别贪多**。必讲的 3 个：

### 概念 1：Function Calling ≠ 「让 AI 写代码调用 API」

学员最大误区：以为 Function Calling 是给 AI 解释「怎么写 requests.get()」。

**真相**：

> LLM **不会**直接调用 API。它只是**返回一段 JSON**，告诉开发者「该调什么、传什么参数」。**真正的执行是开发者写的代码。**

演示代码（手写 1 个 Function Calling）：

```python
# 第一步：开发者定义工具
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "查询天气",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string"}
            }
        }
    }
}]

# 第二步：调 LLM，LLM 不会直接调 API
resp = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "深圳天气"}],
    tools=tools
)

# 第三步：看 LLM 说了什么
print(resp.choices[0].message.tool_calls)
# 输出：[ToolCall(name='get_weather', args='{"city":"深圳"}')]

# 第四步：开发者执行工具
result = call_weather_api("深圳")

# 第五步：把结果回传 LLM，让它生成最终回答
```

**讲师金句**：「LLM 是大脑，工具是手脚。**大脑不会自动长出手脚**，你得给它装上。」

### 概念 2：ReAct = Reason + Act

这是 Agent 循环的标准范式，几乎所有 Agent 框架（LangChain、AutoGen、CrewAI）都基于这个思想。

**为什么叫 ReAct**：

```
Reason（思考）: "用户想知道天气，我需要调 get_weather"
   ↓
Act（行动）: 调用 get_weather(city="深圳")
   ↓
Observation（观察）: 拿到结果 "25°C 晴"
   ↓
Reason（再思考）: "信息够了，可以回答了"
   ↓
Final Answer: "深圳今天 25 度，晴天"
```

**画在白板上**（必备环节）：

```
   ┌──────┐  Think   ┌──────┐  Act    ┌──────┐
   │ LLM  │ ──────── │ Tool │ ──────► │ Result │
   │      │          │ Call │         │       │
   └──┬───┘          └──────┘         └──┬────┘
      ▲                                  │
      │              Observe             │
      └──────────────────────────────────┘
```

让学员**亲手画一遍**这个图，比讲 10 遍都管用。

### 概念 3：工具描述 = Prompt 工程

很多学员写工具描述：

```python
# ❌ 烂描述
@tool
def get_weather(city):
    """获取天气"""
    ...
```

LLM 看了一脸懵：「这玩意儿什么时候用？参数啥意思？」

**正确描述**（讲师要逐字讲）：

```python
@tool
def get_weather(city: str, date: str = "今天") -> str:
    """查询指定城市和日期的天气信息。
    
    适用场景：
    - 用户问"X 天气怎么样"、"X 冷不冷"、"X 下不下雨"
    - 用户问"明天/后天/这周末 X 天气"
    
    Args:
        city: 城市名称，必须是中文，例如"深圳"、"北京"
        date: 日期，可选值"今天"/"明天"/"后天"，或 ISO 格式 "2024-01-15"
    
    Returns:
        天气描述，例如"深圳 25°C 晴"
    """
    ...
```

**讲师要点**：

| 要素 | 必要性 | 说明 |
|------|--------|------|
| **工具名** | 必填 | 动词开头（get/calculate/search） |
| **一句话功能** | 必填 | 让 LLM 知道何时用 |
| **适用场景** | 强烈推荐 | 帮 LLM 决策 |
| **参数说明** | 必填 | 类型 + 含义 + 示例 |
| **返回值说明** | 推荐 | 帮 LLM 理解输出 |
| **使用示例** | 加分 | Few-shot |

## 三、其他概念，根据学员水平选讲

### 概念 4：循环控制（中级学员讲）

**为什么需要控制**：

LLM 不会"自动停止"。它会一直循环调工具、循环调工具……直到：
- Token 耗尽
- 时间超时
- **你的代码强制打断**

**4 道防线**（架构书里有完整代码）：

| 防线 | 设置 |
|------|------|
| 最大迭代次数 | 5-8 次 |
| 总超时时间 | 30 秒 |
| 最大成本 | $0.1/会话 |
| 重复检测 | 连续 3 次相同工具调用就退出 |

### 概念 5：Token 成本（所有学员都要讲）

让学员算一笔账：

```
单次 Agent 循环：假设 5 轮
每轮 Prompt：~2000 Token（含对话历史 + 工具结果）
每轮 LLM 调用：~$0.001（gpt-4o-mini）

单次复杂任务：5 × $0.001 = $0.005
每天 1000 用户 × 5 次任务 = 5000 次 = $25/天 = ¥180/天
```

**讲师金句**：

> "Agent 比普通 ChatGPT **贵 5-10 倍**，因为要循环。**没有成本意识的 Agent 开发，就是在烧钱。**"

## 四、代码讲解：手写 30 分钟 + 框架 30 分钟

### 第一段：手写 ReAct 循环（30 分钟）

**讲师必做**：先手写，**不要直接上 LangChain**。

为什么：

| 手写 | 用框架 |
|------|--------|
| 看到每一步细节 | 黑盒，出了问题不知道在哪 |
| 真正理解 ReAct | 知道原理，框架换了也能写 |
| 调试更容易 | 框架 bug 难定位 |

**关键代码**（架构书有完整版）：

```python
def run_agent(user_input, max_iter=5):
    messages = [{"role": "user", "content": user_input}]
    
    for i in range(max_iter):
        # 1. LLM 决策
        resp = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            tools=TOOLS_SCHEMA
        )
        msg = resp.choices[0].message
        
        # 2. 没有工具调用 = 决策完成
        if not msg.tool_calls:
            return msg.content
        
        # 3. 执行工具
        messages.append(msg)
        for call in msg.tool_calls:
            result = execute_tool(call)
            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": result
            })
    
    return "达到最大迭代次数"
```

**讲师带学员逐行读**，重点解释：

- 为什么 `messages.append(msg)`？——保留 LLM 的决策历史
- 为什么 `tool_call_id` 要传回去？——让 LLM 知道结果对应哪个调用
- 为什么用 `role: "tool"`？——OpenAI 的多角色协议

### 第二段：LangChain 重构（30 分钟）

**讲师话术**：

> "手写是为了理解原理。生产环境用框架——节省 80% 代码量，专注业务逻辑。"

对比代码量：

| 方式 | 代码行数 | 维护性 |
|------|----------|--------|
| 手写 | 60-80 行 | 低（自己处理边界） |
| LangChain | 5-10 行 | 高（社区维护） |

LangChain 版：

```python
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
agent = create_react_agent(llm, [get_weather, calculator])

response = agent.invoke({
    "messages": [HumanMessage(content="深圳明天下不下雨？")]
})
```

**讲师金句**：

> "LangGraph 是 LangChain 的新版本，专门做 Agent。用 `create_react_agent` 一行代码搞定——**但 80% 的 bug 出在'我以为它会处理'的边界情况**。手写过的学员才懂哪里会出问题。"

## 六、必讲的 4 个踩坑点

### 坑 1：工具描述太简略

**现象**：LLM 不知道什么时候用这个工具，结果绕半天不调。

**演示**：故意写一个烂描述，让学员亲眼看到 Agent 决策混乱。

### 坑 2：工具返回复杂对象

```python
# ❌ 错误
@tool
def get_data():
    return {"status": 200, "data": [...]}  # LLM 不会处理 dict

# ✅ 正确
@tool
def get_data():
    data = {"status": 200, "data": [...]}
    return json.dumps(data, ensure_ascii=False)  # 转字符串
```

### 坑 3：参数 schema 不严格

```python
# ❌ 松散
@tool
def get_weather(city, date="今天"):
    ...

# ✅ 严格
@tool
def get_weather(city: str, date: str = "今天") -> str:
    """..."""
```

Pydantic 会帮 IDE 检查类型，但**不帮 LLM**——LLM 看不到类型提示，要靠 docstring。

### 坑 4：没设置 max_iter

**现象**：Agent 死循环，Token 烧光。

**对策**：永远设置 `max_iter=5` 或 `max_iter=8`。

## 八、Q&A 预案

**Q1：Function Calling 和 Tool Use 是一回事吗？**
A：基本是。OpenAI 叫 Function Calling，Anthropic 叫 Tool Use，Google 叫 Function Calling。**协议不同，实现相似**。学一个就会其他。

**Q2：为什么不直接让 LLM 写 Python 代码调 API？**
A：可以，但**不安全**。Function Calling 的核心是 LLM 不执行代码，**只生成意图**，由开发者控制执行边界。

**Q3：怎么让 Agent 学会"用工具 A 而不是 B"？**
A：①工具描述写清楚 ②Few-shot 示例 ③增加"决策"环节（先输出"我要用 X 工具，因为 Y"）。

**Q4：Agent 调错了工具，怎么纠错？**
A：在 Prompt 里加："如果你不确定该用哪个工具，先问用户"。

**Q5：能加 100 个工具吗？**
A：能，但不推荐。**工具超过 20 个，LLM 选择准确率显著下降**。生产环境**用 RAG-as-Tool 模式**——把工具库当文档检索，只把最相关的 5-10 个工具喂给 LLM。
