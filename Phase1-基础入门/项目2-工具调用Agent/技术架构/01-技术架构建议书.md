## 2. 业务诉求与技术挑战

### 2.1 业务诉求

构建一个能**主动调用外部工具**的助手，典型场景：

- "深圳明天下不下雨？→ 调用天气 API"
- "上周我读了哪几篇关于 RAG 的论文？→ 调用本地文件检索"
- "把这 3 个 CSV 合并、去重、按日期排序 → 调用 Python 解释器"
- "帮我在日历上加个会，明天 10 点 → 调用日历 API"

### 2.2 核心挑战

| 挑战 | 表现 | 影响 |
|------|------|------|
| **工具选择** | 用户的请求该用哪个工具？多个工具怎么组合？ | 选错工具，直接答错 |
| **参数抽取** | 工具需要结构化参数，从自然语言怎么抽？ | 参数错了，工具失败 |
| **循环控制** | Agent 可能陷入"思考→调用→失败→再思考→再调用"的死循环 | 烧 Token、用户体验差 |
| **错误恢复** | 工具调用失败（API 超时、参数错误）怎么办？ | 不处理就直接崩 |
| **Token 消耗** | 每轮循环都要调一次 LLM，循环 5 次 = 5 倍成本 | 复杂任务成本爆炸 |

> 💡 **架构核心**：Agent = LLM（决策）+ Tools（执行）+ Loop（迭代）。三者缺一不可。

## 3. 整体架构

### 3.1 Agent 循环（ReAct 范式）

```
        ┌─────────────────────────────────────┐
        │                                     │
        ▼                                     │
   [用户输入]                                  │
        │                                     │
        ▼                                     │
   [LLM 思考] ──── "我需要查天气"                │
        │                                     │
        ├──► 决策: 调用工具？ ─── 否 ──► [返回答案] │
        │                                     │
        ▼ 是                                  │
   [生成工具调用] (function_call)                │
        │                                     │
        ▼                                     │
   [执行工具] (查天气 API)                      │
        │                                     │
        ▼                                     │
   [获得结果] ────────────────┐                │
        │                     │                │
        ▼                     │                │
   [LLM 再思考] ◄─────────────┘                │
        │                                      │
        └──► 继续循环 / 达到 max_iter 退出      │
                                              │
   (循环结束条件: LLM 输出最终答案 / 超过最大迭代)│
        └─────────────────────────────────────┘
```

### 3.2 Function Calling 本质

**核心原理**：

```
┌────────────────────────────────────────────┐
│  1. 开发者定义工具列表 (tools schema)        │
│     - 工具名: get_weather                    │
│     - 描述: 查询指定城市的天气               │
│     - 参数: {city: string, date: string}     │
└──────────────────┬─────────────────────────┘
                   │ 提交给 LLM
                   ▼
┌────────────────────────────────────────────┐
│  2. LLM 不直接回答，而是返回结构化指令        │
│     {                                       │
│       "tool": "get_weather",                │
│       "args": {"city": "深圳", "date": "..."}│
│     }                                       │
└──────────────────┬─────────────────────────┘
                   │ 开发者执行工具
                   ▼
┌────────────────────────────────────────────┐
│  3. 工具执行结果回传给 LLM                   │
│     "深圳明天 25°C，晴"                      │
└──────────────────┬─────────────────────────┘
                   │ LLM 综合信息
                   ▼
┌────────────────────────────────────────────┐
│  4. LLM 生成最终回答                         │
│     "深圳明天 25 度，晴天，适合出门。"        │
└────────────────────────────────────────────┘
```

## 5. 关键技术点设计

### 5.1 工具定义（OpenAI Tool 规范）

**LangChain 方式（推荐基础版）**：

```python
from langchain_core.tools import tool

@tool
def get_weather(city: str, date: str = "今天") -> str:
    """查询指定城市和日期的天气
    
    Args:
        city: 城市名，例如 "深圳"、"北京"
        date: 日期，默认为"今天"，也可传 "明天"、"后天"
    """
    # 真实实现：调用天气 API
    import requests
    url = f"https://wttr.in/{city}?format=j1"
    data = requests.get(url).json()
    return f"{city} {date} 天气：{data['current_condition'][0]['weatherDesc'][0]['value']}"

@tool
def calculator(expression: str) -> str:
    """数学计算器
    
    Args:
        expression: 数学表达式，例如 "2+3*4"、"sqrt(16)"
    """
    import numexpr as ne
    return str(ne.evaluate(expression))
```

**关键设计原则**：

| 原则 | 做法 | 为什么 |
|------|------|--------|
| **描述清晰** | Docstring 要写**人话**，说清楚何时用 | LLM 靠描述判断何时调用 |
| **参数明确** | 类型 + 含义 + 示例 | LLM 才能正确抽参 |
| **返回简洁** | 返回**结构化字符串**，不返回复杂对象 | LLM 只能理解文本 |
| **错误友好** | 失败时返回**可读错误信息**，不抛异常 | LLM 可以基于错误重试 |

### 5.2 Agent 循环实现

**LangChain 0.1+ 推荐用法**：

```python
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent
from langchain_core.messages import HumanMessage

# 1. 定义模型
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# 2. 定义工具
tools = [get_weather, calculator]

# 3. 创建 Agent（一行代码搞定）
agent = create_react_agent(llm, tools)

# 4. 调用
response = agent.invoke({
    "messages": [HumanMessage(content="深圳明天下不下雨？")]
})
print(response["messages"][-1].content)
```

**手写 ReAct 循环（理解原理用）**：

```python
import json
from openai import OpenAI

client = OpenAI()
TOOLS_SCHEMA = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "查询天气",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "城市名"},
                    "date": {"type": "string", "description": "日期"}
                },
                "required": ["city"]
            }
        }
    }
]

def run_agent(user_input: str, max_iter: int = 5):
    messages = [{"role": "user", "content": user_input}]
    
    for i in range(max_iter):
        # 1. LLM 决策
        resp = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            tools=TOOLS_SCHEMA
        )
        msg = resp.choices[0].message
        
        # 2. 判断是否要调用工具
        if not msg.tool_calls:
            return msg.content  # 决策完成
        
        # 3. 执行工具
        messages.append(msg)  # 保留 LLM 的工具调用决策
        for call in msg.tool_calls:
            args = json.loads(call.function.arguments)
            if call.function.name == "get_weather":
                result = get_weather(**args)
            # ... 其他工具
            
            # 4. 把工具结果加回 messages
            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": result
            })
    
    return "达到最大迭代次数"
```

**讲师必讲**：手写一遍循环，**理解原理**；再用 LangChain **提高效率**。

### 5.3 工具选择失败的对策

**场景**：用户问「深圳怎么样」，模型不知道该用 `get_weather` 还是别的。

**对策 3 把斧**：

#### 对策 1：清晰的角色定义

```python
SYSTEM_PROMPT = """你是天气助手。

工具说明：
- get_weather: 查询天气，城市 + 日期
- get_time: 查询时间，城市（可选）

规则：
1. 用户问天气，调用 get_weather
2. 用户问时间，调用 get_time
3. 不要编造天气信息
"""
```

#### 对策 2：Few-shot 示例

```python
messages = [
    {"role": "system", "content": SYSTEM_PROMPT},
    {"role": "user", "content": "北京今天冷不冷？"},
    {"role": "assistant", "content": None, "tool_calls": [...]},
    {"role": "tool", "content": "北京 5°C，阴"},
    {"role": "assistant", "content": "北京今天 5 度，阴天，挺冷的。"},
    {"role": "user", "content": user_input}
]
```

#### 对策 3：日志 + 人工干预

```python
import logging
logging.basicConfig(level=logging.INFO)

# 每次工具调用都打日志
logging.info(f"调用工具: {tool_name}, 参数: {args}, 结果: {result}")
```

让学员**看到每一步决策**，便于调试。

### 5.4 循环控制（防死循环）

```python
class AgentConfig:
    max_iterations: int = 5      # 最大迭代次数
    timeout_seconds: int = 30    # 总超时时间
    max_cost_usd: float = 0.1    # 单次会话最大成本
```

**防死循环的 4 道防线**：

| 防线 | 实现 | 触发条件 |
|------|------|----------|
| 迭代次数限制 | `for i in range(max_iter)` | 达到 `max_iter` |
| 时间限制 | `if time.time() - start > timeout: break` | 超过 30s |
| 成本限制 | 累计 Token 计算 | 超过 $0.1 |
| 重复检测 | 检测连续 3 次相同工具调用 | 立即退出 |

### 5.5 错误处理

```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(min=1, max=10))
def call_tool_with_retry(tool_func, **kwargs):
    try:
        return tool_func(**kwargs)
    except Exception as e:
        # 把错误返回给 LLM，让它决定是否重试
        return f"工具执行失败: {str(e)}。请换一种方式或调整参数。"
```

**关键设计**：

- 工具失败**不要直接抛异常给用户**
- 把错误**作为信息返回给 LLM**，让 LLM 决策下一步
- LLM 可能换个工具、换参数、或向用户解释

## 7. 常见坑与对策

| 坑 | 表现 | 对策 |
|----|------|------|
| **工具描述太简略** | "计算工具" → LLM 不知道什么时候用 | 描述要写"何时用 + 怎么用 + 示例" |
| **参数 schema 错误** | LLM 抽不出参数 | 用 pydantic 定义 schema，让 IDE 帮你检查 |
| **Agent 死循环** | 来回调同一个工具 | 加迭代次数 + 重复检测 |
| **工具返回复杂对象** | 返回 dict/list，LLM 困惑 | 永远返回**字符串**，自己 format 一下 |
| **Token 爆炸** | 工具结果太长，Prompt 膨胀 | 工具结果截断（>2000 字就摘要） |
| **时间格式混乱** | "明天"、"下周三"、"2024-01-15" 混用 | 工具内部统一转 ISO 格式 |
