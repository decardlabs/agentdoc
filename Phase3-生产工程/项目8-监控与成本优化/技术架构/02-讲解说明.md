## 一、开场引导：Agent 翻车了你不知道

老板，这个项目**不讲概念，讲救命**。

开场可以这样讲：

> "你做了一个 Agent 上线了。用户说'回答不对'——你打开日志，全是 `print('ok')`，一行有用的信息都没有。
>
> 你去查 OpenAI 后台——这个月花了 $500，上周才 $50。哪个用户花的？不知道。哪天突然涨的？不知道。哪个模型最费钱？不知道。
>
> 老板问你这 $500 花哪了，你说不出来。这就是**没监控的代价**——不是钱的问题，是**你不知道钱花哪了**。"

**让学员感受痛点**（现场演示）：

```python
# 一个没监控的 Agent
from openai import OpenAI
client = OpenAI()

def ask_agent(question: str) -> str:
    response = client.chat.completions.create(
        model="gpt-4o",  # 用的最贵的模型
        messages=[{"role": "user", "content": question}]
    )
    return response.choices[0].message.content

# 连续调 100 次
for i in range(100):
    result = ask_agent("帮我写一首诗")
    print(f"第 {i+1} 次: {result[:20]}...")

# 现在问你：
# 1. 刚才花了多少钱？ → 不知道
# 2. 平均延迟多少？ → 不知道
# 3. 有几次失败？ → 不知道
# 4. 用了什么模型？ → gpt-4o，但为什么要用最贵的写诗？
```

**讲师金句**：

> "没监控的系统 = 闭眼开车。你不知道速度、不知道油量、不知道前面是不是悬崖。**出事是早晚的，不是会不会的问题**。"

**埋钩子**：

> "这个项目做完，你会有 4 样东西：**追踪（哪里错了）、指标（多快多慢）、日志（发生了什么）、成本（花了多少钱）**。有了这 4 样，Agent 出问题你 5 分钟就能定位。"

## 二、4 个核心概念，必须讲透

### 概念 1：Tracing = 请求的 GPS 轨迹

学员最困惑的：「追踪和日志有什么区别？不都是记录吗？」

**关键区别**：

| 维度 | 日志 (Logging) | 追踪 (Tracing) |
|------|---------------|---------------|
| 粒度 | 单条事件 | 一条请求的完整链路 |
| 关系 | 独立的，没有关联 | 有父子关系的 Span Tree |
| 问的问题 | "发生了什么？" | "哪里慢了？哪里错了？" |
| 例子 | `2025-06-14 LLM 调用成功` | 用户请求 → 意图识别 → 工具调用 → 生成回答 |

**演示**（必做）：

```
一次 Agent 请求的 Trace:

[Trace: trace_abc123] 用户: "深圳明天天气怎么样？"
  ├── [Span 1] identify_intent (1.2s, 150 tokens)
  │   └── 意图: 查天气, 城市: 深圳, 日期: 明天
  ├── [Span 2] call_tool: get_weather (0.8s)
  │   └── 结果: 深圳 25°C, 晴
  └── [Span 3] generate_answer (1.5s, 280 tokens)
      └── 结果: "深圳明天 25 度，晴天..."

总耗时: 3.5s | 总成本: $0.004 | 总 Token: 430
```

> **讲师要点**：看到这棵树，你就知道**每一步花了多少时间、多少 Token**。如果总耗时 10 秒，一看就知道是 Span 3 慢了。

**对比没 Tracing 的体验**：

```
你只有一行日志:
2025-06-14 10:30:45 INFO Agent call completed

出问题了——用户说 10 秒才返回。你从这行日志能看出什么？什么都看不出来。
```

### 概念 2：Metrics = 系统的仪表盘

学员容易把 Metrics 和 Logging 混为一谈。

**比喻**：

> 开车的时候，速度表、油量表、转速表 = Metrics。行车记录仪 = Logging。
> Metrics 告诉你"现在 120km/h"，Logging 告诉你"3:45 经过了一个红灯"。

**三种 Metrics 类型**：

```python
# Counter: 只增不减（像汽车里程表）
llm_call_total = Counter("llm_call_total", "Total calls", ["model", "status"])
llm_call_total.labels(model="gpt-4o-mini", status="success").inc()

# Gauge: 可增可减（像油量表）
active_sessions = Gauge("active_sessions", "Active sessions")
active_sessions.inc()   # 用户上线
active_sessions.dec()   # 用户下线

# Histogram: 分布统计（像速度区间统计）
llm_call_duration = Histogram("llm_call_duration_seconds", "Latency", ["model"])
llm_call_duration.labels(model="gpt-4o-mini").observe(1.2)  # 记录 1.2s
```

**讲师必讲**：

> "为什么需要 Histogram 而不是 Gauge？——因为**平均值是骗人的**。10 次调用，9 次 0.5s，1 次 50s，平均值 5.45s。但你告诉用户'平均 5.5 秒'，90% 的用户体验其实是好的。
>
> Histogram 给你 P50/P95/P99——**P95 延迟 50s 意味着 5% 的用户体验极差**。这才是你需要优化的。"

**P50/P95/P99 的直观解释**：

```
100 次调用按延迟排序:

  #1  0.3s  ─┐
  #2  0.4s   │ P50 = 0.8s → 一半的请求 < 0.8s
  ...        │
  #50 0.8s  ─┘
  ...
  #95 3.2s  ── P95 = 3.2s → 95% 的请求 < 3.2s
  ...
  #99 8.5s  ── P99 = 8.5s → 99% 的请求 < 8.5s
  #100 50s  ── 最差的 1%

你的 SLO 定 P95 < 5s → 看这组数据，达标了。
如果 P95 = 8s → 有 5% 的用户体验很差，得修。
```

### 概念 3：Logging = 事件的黑匣子

**为什么必须用结构化日志**：

```python
# ❌ 文本日志（没法查询）
print("2025-06-14 10:30:45 - LLM call succeeded for user_001 with gpt-4o-mini, cost $0.003, latency 1200ms")

# ✅ JSON 日志（可以查询、过滤、聚合）
logger.info("LLM call completed", extra={
    "user_id": "user_001",
    "model": "gpt-4o-mini",
    "cost_usd": 0.003,
    "latency_ms": 1200
})
# 输出:
# {"timestamp":"2025-06-14T10:30:45Z","level":"INFO","message":"LLM call completed","user_id":"user_001","model":"gpt-4o-mini","cost_usd":0.003,"latency_ms":1200}
```

**查询对比**：

```bash
# 文本日志：想找 user_001 的所有调用
grep "user_001" app.log | grep "cost" | awk '{print $NF}'
# 能跑，但很脆弱——格式变了就挂

# JSON 日志：同样的事
cat app.log | jq 'select(.user_id == "user_001") | .cost_usd'
# 稳定、精确、不受文本格式变化影响
```

**讲师金句**：

> "文本日志是人看的，JSON 日志是机器看的。生产环境日志是给机器看的——**你不会一条一条读日志，你要用工具查**。"

### 概念 4：Cost = 钱包的实时监控

**Token 成本不是线性的**——模型不同，价格差 10-100 倍。

**让学员算一笔账**：

```python
# 同一个请求，不同模型的成本差异
prompt_tokens = 2000
completion_tokens = 500

models = {
    "gpt-4o-mini":      {"input": 0.15,  "output": 0.60},
    "gpt-4o":           {"input": 2.50,  "output": 10.00},
    "claude-3.5-sonnet": {"input": 3.00,  "output": 15.00},
}

for model, pricing in models.items():
    cost = (prompt_tokens / 1e6) * pricing["input"] + \
           (completion_tokens / 1e6) * pricing["output"]
    print(f"{model:20s}: ${cost:.4f}")

# gpt-4o-mini      : $0.0006
# gpt-4o           : $0.0100
# claude-3.5-sonnet : $0.0135

# gpt-4o 是 mini 的 16.7 倍！
```

**讲师要点**：

> "一个用户一天问 100 个问题，用 gpt-4o 一天 $1，用 gpt-4o-mini 一天 $0.06。一个月差 $28。
>
> 50 个用户一个月差 $1400。**模型选择就是成本控制**——但前提是你得先监控，否则你不知道谁在用什么模型。"

### Demo 2：自建调用日志 + 成本计算（25 分钟）

**LangSmith 是好的，但它不给你数据——你没法用 SQL 查。自建日志是必须的**。

```python
# demo2_call_logger.py
import sqlite3
import time
import json
from contextlib import contextmanager
from openai import OpenAI

# 定价表
MODEL_PRICING = {
    "gpt-4o-mini":      {"input": 0.15,  "output": 0.60},
    "gpt-4o":           {"input": 2.50,  "output": 10.00},
    "claude-3.5-sonnet": {"input": 3.00,  "output": 15.00},
    "claude-3-haiku":   {"input": 0.25,  "output": 1.25},
    "deepseek-chat":    {"input": 0.14,  "output": 0.28},
}
# 注意：以上价格为文档编写时参考值。模型定价会持续调整（如 GPT-4o 已从 $5/M 降至 $2.50/M）。
# 教学时提醒学员以 openai.com/pricing 和 anthropic.com/pricing 最新价格为准。
# 建议生产环境通过各厂商 API 动态获取最新价格，而非硬编码。
}


class LLMCallLogger:
    def __init__(self, db_path: str = "llm_logs.db"):
        self.db_path = db_path
        self._init_db()

    def _init_db(self):
        with sqlite3.connect(self.db_path) as conn:
            conn.execute("""
                CREATE TABLE IF NOT EXISTS llm_calls (
                    id INTEGER PRIMARY KEY AUTOINCREMENT,
                    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
                    user_id TEXT DEFAULT 'anonymous',
                    session_id TEXT,
                    model TEXT NOT NULL,
                    prompt_tokens INTEGER DEFAULT 0,
                    completion_tokens INTEGER DEFAULT 0,
                    total_tokens INTEGER DEFAULT 0,
                    cost_usd REAL DEFAULT 0,
                    latency_ms INTEGER DEFAULT 0,
                    status TEXT DEFAULT 'success',
                    error_message TEXT,
                    request_preview TEXT,
                    response_preview TEXT
                )
            """)

    def calculate_cost(self, model: str, prompt_tokens: int, completion_tokens: int) -> float:
        if model not in MODEL_PRICING:
            return 0.0
        p = MODEL_PRICING[model]
        cost = (prompt_tokens / 1_000_000) * p["input"] + \
               (completion_tokens / 1_000_000) * p["output"]
        return round(cost, 6)

    @contextmanager
    def log_call(self, model: str, user_id: str = "anonymous", session_id: str = None):
        log_data = {"start_time": time.time()}
        try:
            yield log_data
            status = "success"
            error_msg = None
        except Exception as e:
            status = "error"
            error_msg = str(e)
            raise
        finally:
            latency_ms = int((time.time() - log_data["start_time"]) * 1000)
            prompt_tokens = log_data.get("prompt_tokens", 0)
            completion_tokens = log_data.get("completion_tokens", 0)
            cost = self.calculate_cost(model, prompt_tokens, completion_tokens)

            with sqlite3.connect(self.db_path) as conn:
                conn.execute("""
                    INSERT INTO llm_calls
                    (user_id, session_id, model, prompt_tokens, completion_tokens,
                     total_tokens, cost_usd, latency_ms, status, error_message,
                     request_preview, response_preview)
                    VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
                """, (
                    user_id, session_id, model,
                    prompt_tokens, completion_tokens,
                    prompt_tokens + completion_tokens,
                    cost, latency_ms, status, error_msg,
                    log_data.get("request_preview", "")[:2000],
                    log_data.get("response_preview", "")[:2000]
                ))


# 使用：封装 OpenAI 客户端
class MonitoredClient:
    def __init__(self, logger: LLMCallLogger):
        self.openai = OpenAI()
        self.logger = logger

    def chat(self, messages, model="gpt-4o-mini", user_id="anonymous", **kwargs):
        with self.logger.log_call(model, user_id) as log:
            log["request_preview"] = messages[-1]["content"] if messages else ""
            response = self.openai.chat.completions.create(
                model=model, messages=messages, **kwargs
            )
            log["prompt_tokens"] = response.usage.prompt_tokens
            log["completion_tokens"] = response.usage.completion_tokens
            log["response_preview"] = response.choices[0].message.content[:2000]
            return response.choices[0].message.content


# 测试
logger = LLMCallLogger()
client = MonitoredClient(logger)

# 模拟不同用户、不同模型
client.chat([{"role": "user", "content": "Hello"}], model="gpt-4o-mini", user_id="user_001")
client.chat([{"role": "user", "content": "Explain RAG"}], model="gpt-4o", user_id="user_002")

# 查看记录
import sqlite3
with sqlite3.connect("llm_logs.db") as conn:
    rows = conn.execute("SELECT user_id, model, total_tokens, cost_usd, latency_ms FROM llm_calls").fetchall()
    for row in rows:
        print(f"用户: {row[0]}, 模型: {row[1]}, Token: {row[2]}, 成本: ${row[3]:.4f}, 延迟: {row[4]}ms")
```

**讲师要点**：

> "看，每次调用都有记录了。user_002 用 gpt-4o，成本是 user_001 的 16 倍——**这就是你需要监控的原因**。"

### Demo 3：Prometheus + Grafana 监控（25 分钟）

```python
# demo3_prometheus.py
from prometheus_client import Counter, Histogram, Gauge, start_http_server
import time
import random

# 定义指标
llm_call_total = Counter(
    "llm_call_total",
    "Total LLM calls",
    ["model", "status"]
)

llm_call_duration = Histogram(
    "llm_call_duration_seconds",
    "LLM call duration",
    ["model"],
    buckets=[0.5, 1.0, 2.0, 5.0, 10.0, 30.0]
)

active_sessions = Gauge("active_sessions", "Active sessions")

# 启动 Metrics 端口
start_http_server(8000)
print("Prometheus metrics server started on :8000/metrics")

# 模拟流量
models = ["gpt-4o-mini", "gpt-4o", "claude-3.5-sonnet"]
active_sessions.set(5)

for i in range(100):
    model = random.choice(models)
    is_error = random.random() < 0.05  # 5% 错误率
    latency = random.lognormvariate(0.5, 0.8)  # 模拟真实延迟分布

    # 记录指标
    status = "error" if is_error else "success"
    llm_call_total.labels(model=model, status=status).inc()
    llm_call_duration.labels(model=model).observe(latency)

    time.sleep(0.5)

# 验证: curl http://localhost:8000/metrics
# 然后配置 Prometheus 来 scrape 这个端点
```

### Demo 4：成本 Dashboard（15 分钟）

```python
# demo4_cost_dashboard.py
import streamlit as st
import sqlite3
import pandas as pd
import plotly.express as px
from datetime import datetime, timedelta

st.set_page_config(page_title="LLM Cost Monitor", layout="wide")
st.title("LLM Cost Monitor")

@st.cache_data(ttl=60)
def load_data():
    conn = sqlite3.connect("llm_logs.db")
    df = pd.read_sql("SELECT * FROM llm_calls ORDER BY timestamp DESC", conn)
    df['timestamp'] = pd.to_datetime(df['timestamp'])
    return df

df = load_data()

if len(df) == 0:
    st.warning("No data yet. Run demo2_call_logger.py first.")
    st.stop()

# 核心指标
col1, col2, col3, col4 = st.columns(4)
with col1:
    st.metric("Total Calls", f"{len(df):,}")
with col2:
    st.metric("Total Cost", f"${df['cost_usd'].sum():.2f}")
with col3:
    st.metric("Total Tokens", f"{df['total_tokens'].sum():,}")
with col4:
    error_rate = (df['status'] == 'error').mean() * 100
    st.metric("Error Rate", f"{error_rate:.1f}%")

# 成本趋势
st.subheader("Daily Cost Trend")
daily = df.groupby(df['timestamp'].dt.date).agg({'cost_usd': 'sum', 'id': 'count'}).reset_index()
daily.columns = ['date', 'cost_usd', 'calls']
fig = px.bar(daily, x='date', y='cost_usd', title="Daily Cost (USD)")
st.plotly_chart(fig, use_container_width=True)

# 模型分布
st.subheader("Cost by Model")
model_cost = df.groupby('model').agg({'cost_usd': 'sum', 'id': 'count'}).reset_index()
fig = px.pie(model_cost, values='cost_usd', names='model', title="Cost Distribution")
st.plotly_chart(fig, use_container_width=True)

# Top 用户
st.subheader("Top Users by Cost")
user_cost = df.groupby('user_id').agg({
    'cost_usd': 'sum', 'total_tokens': 'sum', 'id': 'count'
}).sort_values('cost_usd', ascending=False).head(10)
st.dataframe(user_cost)

# 最近调用
st.subheader("Recent Calls")
st.dataframe(df[['timestamp', 'user_id', 'model', 'total_tokens', 'cost_usd', 'latency_ms', 'status']].head(20))
```


**讲师要点**：

> "异常检测不需要复杂算法。3-sigma 规则 + 环比对比就能解决 80% 的问题。**别上来就想搞 ML 异常检测——先让简单的方案跑起来**。"

## 四、调试技巧：怎么从监控数据找到 Bug

### 招数 1：从 Grafana 发现问题，从 Trace 定位问题

**典型场景**：Grafana 显示错误率从 0.5% 涨到 5%。

```
Step 1: Grafana → 看到错误率飙升
Step 2: Prometheus 查询 → 确认是哪个模型出错
  sum by (model) (rate(llm_call_total{status="error"}[5m]))
  → gpt-4o 的错误率 15%
Step 3: LangSmith → 过滤 gpt-4o 的 error traces
  → 发现都是 RateLimitError
Step 4: 查看具体 Trace → 请求太频繁，需要加限流
```

**讲师金句**：

> "监控是'发现了问题'，追踪是'找到了原因'。**两条腿走路**。"

### 招数 2：用 SQL 分析成本异常

```sql
-- 哪个用户今天最费钱？
SELECT user_id, SUM(cost_usd) as total_cost, COUNT(*) as calls
FROM llm_calls
WHERE DATE(timestamp) = CURRENT_DATE
GROUP BY user_id
ORDER BY total_cost DESC
LIMIT 10;

-- 哪个时段成本最高？
SELECT strftime('%H', timestamp) as hour,
       SUM(cost_usd) as total_cost,
       COUNT(*) as calls
FROM llm_calls
WHERE DATE(timestamp) = CURRENT_DATE
GROUP BY hour
ORDER BY total_cost DESC;

-- 有没有单次调用特别贵的？
SELECT id, user_id, model, total_tokens, cost_usd, latency_ms
FROM llm_calls
WHERE cost_usd > 0.05  -- 单次超过 5 美分
ORDER BY cost_usd DESC
LIMIT 20;
```

### 招数 3：对比不同时间段的指标

```python
# 用 PromQL 做同比对比
# 今天 vs 昨天同时段的 QPS
query_today = 'sum(rate(llm_call_total[1h]))'
query_yesterday = 'sum(rate(llm_call_total[1h] offset 24h))'

# 增长率
query_growth = """
  (sum(rate(llm_call_total[1h]))
   /
   sum(rate(llm_call_total[1h] offset 24h)))
   * 100
"""
# 值 = 150 → 今天比昨天同时段增长 50%
# 值 = 300 → 今天是昨天的 3 倍！异常！
```

### 招数 4：从日志提取错误模式

```bash
# 找出所有错误类型
```

## 五、6 个踩坑点

### 坑 1：告警疲劳

**现象**：一天收到 500 条告警，全是低优先级。你把群设为免打扰。真正重要的告警来了，你看不到。

**原因**：告警阈值设太低，什么都告警。

**对策**：

| 原则 | 做法 | 效果 |
|------|------|------|
| **分级** | Critical / Warning / Info | 人只看 Critical |
| **去重** | 5 分钟内相同告警只发 1 条 | 减少 80% 噪音 |
| **聚合** | 同组告警合并成 1 条 | 减少 90% 噪音 |
| **可操作性** | 告警必须告诉人"该做什么" | 不发无意义的告警 |

**讲师金句**：

> "一条告警如果没人看，不如不发。**告警的敌人不是延迟，是疲劳**。"

### 坑 2：Metric 爆炸

**现象**：给每个用户都建了一个 Metric label → `llm_call_total{user_id="user_001"}`, `llm_call_total{user_id="user_002"}`, ... 10 万个用户 = 10 万个时间序列。

**后果**：Prometheus 内存爆炸，查询超时。

**对策**：

```python
# ❌ 高基数 label（每个用户一个序列）
llm_call_total = Counter("llm_call_total", "calls", ["model", "status", "user_id"])

# ✅ 低基数 label（用户信息存在日志里，不在 Metric 里）
llm_call_total = Counter("llm_call_total", "calls", ["model", "status"])
# 用户级成本用 SQL 查，不用 Prometheus
```

**规则**：Prometheus label 的基数不超过 50。用户 ID、Session ID 这种高基数值，放日志，不放 Metrics。

### 坑 3：存储爆炸

**现象**：每次 LLM 调用都存完整的 request/response，一个月数据库 50GB。

**原因**：一次 GPT-4o 调用的 request+response 平均 5KB，10000 次/天 = 50MB/天 = 1.5GB/月。加上工具结果，3-5 倍。

**对策**：

| 策略 | 做法 | 节省 |
|------|------|------|
| **截断** | request/response 只存前 2000 字符 | 70% |
| **采样** | 成功的调用 10% 采样，失败的全存 | 90% |
| **归档** | 30 天后删原始数据，只保留聚合 | 95% |
| **压缩** | JSONB 列用 TOAST 压缩 | 50% |

```python
# 采样策略
import random

def should_log_full(status: str, sample_rate: float = 0.1) -> bool:
    """决定是否存完整日志"""
    if status == "error":
        return True  # 错误的全存
    return random.random() < sample_rate  # 成功的 10% 采样
```

### 坑 4：监控本身成为瓶颈

**现象**：高频调用时，同步写数据库把响应时间从 1s 拉到 3s。

**对策**：

```python
import asyncio
from asyncio import Queue

class AsyncBatchLogger:
    """异步批量日志写入器"""

    def __init__(self, batch_size: int = 100, flush_interval: float = 1.0):
        self.queue = Queue()
        self.batch_size = batch_size
        self.flush_interval = flush_interval

    async def log(self, data: dict):
        """非阻塞写入队列"""
        await self.queue.put(data)

    async def writer_loop(self):
        """后台写入协程"""
        while True:
            batch = []
            try:
                # 等待第一个元素
                item = await asyncio.wait_for(self.queue.get(), timeout=self.flush_interval)
                batch.append(item)

                # 尝试收集更多（非阻塞）
                while len(batch) < self.batch_size:
                    try:
                        item = self.queue.get_nowait()
                        batch.append(item)
                    except asyncio.QueueEmpty:
                        break

            except asyncio.TimeoutError:
                pass

            if batch:
                await self._batch_insert(batch)

    async def _batch_insert(self, batch: list[dict]):
        """批量写入数据库"""
        import sqlite3
        with sqlite3.connect("llm_logs.db") as conn:
            conn.executemany("""
                INSERT INTO llm_calls
                (user_id, model, prompt_tokens, completion_tokens,
                 total_tokens, cost_usd, latency_ms, status)
                VALUES (?, ?, ?, ?, ?, ?, ?, ?)
            """, [
                (d.get("user_id"), d.get("model"), d.get("prompt_tokens"),
                 d.get("completion_tokens"), d.get("total_tokens"),
                 d.get("cost_usd"), d.get("latency_ms"), d.get("status"))
                for d in batch
            ])
```

**讲师金句**：

> "监控不能比业务还慢。**日志必须异步，指标必须轻量**。"

### 坑 5：漏采样导致关键数据丢失

**现象**：采样率 10%，结果一个关键错误被漏掉了，事后查不到。

**对策**：

```python
def should_sample(status: str, cost_usd: float, latency_ms: int) -> bool:
    """智能采样：重要的全存，不重要的采样"""
    # 1. 错误全存
    if status == "error":
        return True

    # 2. 高成本调用全存
    if cost_usd > 0.01:
        return True

    # 3. 高延迟调用全存
    if latency_ms > 10000:
        return True

    # 4. 其余 10% 采样
    return random.random() < 0.1
```

**规则**：**错误、高成本、高延迟，这三类必须全量记录**。只有正常调用才采样。

### 坑 6：成本计算与实际账单对不上

**现象**：你算出来一个月 $100，OpenAI 账单 $130。

**原因**：

| 原因 | 影响幅度 | 对策 |
|------|---------|------|
| 模型版本差异（gpt-4o vs gpt-4o-2024-08-06） | 5-10% | 统一用 API 返回的 model 字段 |
| 缓存 Token 不同价 | 10-50% | 区分 cached_input 和 input |
| 批处理 API 半价 | 50% | 区分 batch 和 realtime |
| Token 计数方法差异 | 1-3% | 用 API 返回的 usage 字段，不要自己数 |

**对账脚本**：

```python
def reconcile_with_provider(db_cost: float, provider_cost: float) -> dict:
    """与供应商账单对账"""
    diff_pct = abs(db_cost - provider_cost) / provider_cost * 100

    return {
        "db_cost": round(db_cost, 2),
        "provider_cost": round(provider_cost, 2),
        "diff_usd": round(abs(db_cost - provider_cost), 2),
        "diff_pct": round(diff_pct, 1),
        "status": "OK" if diff_pct < 10 else "MISMATCH"
    }

# 示例
result = reconcile_with_provider(db_cost=98.5, provider_cost=130.0)
# diff_pct = 24.2%, status = MISMATCH → 需要排查
```

**讲师金句**：

> "成本计算允许 10% 误差。**超过 10% 就是有 bug，必须排查**。最常见的 bug 是漏了 cached token 的价格差异。"

## 六、Q&A 预案

**Q1：LangSmith 和自建日志，必须都做吗？**

A：基础版做 LangSmith 就行（3 分钟接入，立刻能看到数据）。但**生产环境必须自建**——LangSmith 是 SaaS，你的数据在别人服务器上，而且免费额度只有 5K traces/月。自建日志 = 数据在你手里，想怎么查怎么查。

**Q2：Prometheus 和 Grafana 好复杂，能不能用更简单的方案？**

A：基础版用 Streamlit Dashboard 就够了。Prometheus + Grafana 是**生产版**的标配，但学习曲线确实陡。建议：**先用 Streamlit 跑通，再升级到 Grafana**。

**Q3：采样率怎么定？**

A：看你的调用量和预算。1000 次/天以下不用采样。10000 次/天用 50% 采样。10 万次/天用 10% 采样。**记住：错误、高成本、高延迟必须全量记录**。

**Q4：告警发到哪里？钉钉、飞书、邮件选哪个？**

A：团队用什么就发哪里。钉钉/飞书适合实时看，邮件适合日报。**不要同时发所有渠道——一个渠道就够了**，否则又是告警疲劳。

**Q5：异常检测用 ML 还是统计方法？**

A：**先统计，后 ML**。3-sigma + 环比能解决 80% 的问题。只有数据量极大、模式很复杂时才需要 ML。不要上来就搞 LSTM 异常检测——**你连基线都没有，检测什么？**

**Q6：监控数据存多久？**

A：原始数据 30 天，聚合数据永久。**30 天前的原始数据如果还留着，你大概率也不会看**。但成本聚合（每日/每用户/每模型）必须永久保留——这是对账和趋势分析的基础。

**Q7：这个项目的监控能复用到其他项目吗？**

A：**必须复用**。MonitoredClient、LLMCallLogger、Prometheus 指标定义，这些都是**基础设施**，不是项目特有的。项目 9（安全与评估）会直接基于这个监控框架做安全审计。

**Q8：生产环境怎么保证监控本身不挂？**

A：三条原则：
1. 监控组件独立部署，不和业务服务混部
2. 日志写入用异步队列，写入失败不影响业务
3. Prometheus 和 Grafana 做高可用（至少 2 副本）
