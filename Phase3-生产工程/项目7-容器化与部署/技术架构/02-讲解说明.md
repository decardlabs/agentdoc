## 一、开场引导：让学员感受"本地能跑不等于生产能用"

老板，这个项目跟前面 6 个项目的区别在于——**前面都是写代码，这个项目是"让你的代码活在线上"**。

开场建议直接"炸"一下学员：

> "你们项目 6 的多 Agent 系统跑起来了吧？好，现在我问三个问题：
>
> 1. 你把代码发给同事，他能在自己电脑上跑起来吗？（不能，Python 版本不对）
> 2. 你想把系统部署到一个公网地址让别人用，你怎么做？（不知道）
> 3. 你半夜三点代码炸了，你怎么快速恢复？（手动 ssh 上去重启？）
>
> **这三个问题就是容器化要解决的。**"

然后直接演示一下"本地 vs 生产"的差异：

```bash
# 在学员的 Mac 上（本地）
python app/main.py  # 跑得好好的

# 在一台干净的 Linux 服务器上（生产）
python app/main.py
# ModuleNotFoundError: No module named 'autogen_agentchat'
# 然后你 pip install，发现 Python 版本不对
# 然后你换 Python 版本，发现某个 C 扩展编译失败
# 然后你折腾了 2 个小时，还没跑起来
```

> **讲师金句**："这就是容器化之前的状态——每台机器都是一个雷区。Docker 的价值就是：**在雷区上铺一层水泥，到处都是平地。**"

**这个项目有 3 个不可替代的价值**：

1. **从"能跑"到"能上线"**——这是区分"写代码的人"和"工程师"的分水岭
2. **掌握 CI/CD 思维**——push 代码自动测试、自动构建、自动部署，这是工业标准
3. **理解生产环境**——端口、防火墙、HTTPS、日志、监控，这些东西写代码时根本不关心，但上线后一个都不能少

## 二、4 个核心概念，必须讲透

### 概念 1：容器 = "带运行环境的代码快递包"

很多学员搞不清容器和虚拟机的区别。

**直接上对比图**：

```
虚拟机:
┌─────────────────────────────────────────┐
│          宿主机操作系统                    │
│  ┌──────────────┐  ┌──────────────┐     │
│  │  Guest OS A  │  │  Guest OS B  │     │
│  │  (完整系统)   │  │  (完整系统)   │     │
│  │  ┌────────┐  │  │  ┌────────┐  │     │
│  │  │  应用  │  │  │  │  应用  │  │     │
│  │  └────────┘  │  │  └────────┘  │     │
│  └──────────────┘  └──────────────┘     │
│  启动: 分钟级 | 大小: GB 级 | 资源重    │
└─────────────────────────────────────────┘

容器:
┌─────────────────────────────────────────┐
│          宿主机操作系统                    │
│  ┌──────────────────────────────────┐   │
│  │  Docker Engine（共享宿主机内核）    │   │
│  │  ┌──────┐  ┌──────┐  ┌──────┐    │   │
│  │  │App A │  │App B │  │App C │    │   │
│  │  │+ 依赖 │  │+ 依赖 │  │+ 依赖 │    │   │
│  │  └──────┘  └──────┘  └──────┘    │   │
│  │  启动: 秒级 | 大小: MB 级 | 轻量    │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**关键区别**：

| 维度 | 虚拟机 | 容器 |
|------|--------|------|
| 启动时间 | 分钟级 | **秒级** |
| 资源占用 | GB 级 | **MB 级** |
| 性能 | 有损耗 | **接近原生** |
| 隔离性 | 强（独立内核） | 弱（共享内核） |
| 使用场景 | 需要完全隔离的环境 | **应用部署** |

**讲师要点**：容器不是虚拟机的替代品，而是"进程级隔离"。你把 Python 应用 + 它需要的所有依赖打包在一起，这就是容器。

**镜像 vs 容器**的比喻：

> "镜像是菜谱，容器是按菜谱做出来的菜。同一个菜谱可以做很多份菜。你修改了菜谱（重新 build 镜像），旧菜不会自动变。"

### 概念 2：Docker Compose = "一键拉起整套系统"

学员写完一个 Agent 服务时，需要 Redis、Postgres、Chroma 等依赖。以前你得一个一个 `docker run`，命令长到离谱：

```bash
# 不用 Compose 的噩梦（手敲 500 字命令）
docker run -d --name redis -p 6379:6379 -v redis-data:/data \
  redis:7-alpine redis-server --appendonly yes

docker run -d --name postgres -p 5432:5432 -e POSTGRES_DB=agentdb \
  -e POSTGRES_USER=agent -e POSTGRES_PASSWORD=agent123 \
  -v postgres-data:/var/lib/postgresql/data postgres:16-alpine

docker run -d --name chroma -p 8001:8000 \
  -v chroma-data:/chroma/chroma chromadb/chroma:latest

docker run -d --name agent -p 8000:8000 \
  --link redis --link postgres --link chroma \
  -e REDIS_URL=redis://redis:6379/0 \
  -e DATABASE_URL=postgresql://agent:agent123@postgres:5432/agentdb \
  -v ./logs:/app/logs multi-agent:v1.0

# 停止呢？一个个 docker stop
# 清理呢？一个个 docker rm
# 重启呢？完蛋，顺序不能乱...
```

**用 Compose 之后**：

```bash
docker compose up -d    # 一条命令拉起全套
docker compose down      # 一条命令全停
docker compose logs -f   # 一条命令看日志
docker compose ps        # 一条命令看状态
```

> **讲师金句**："Compose 就是给你的 Docker 加了一个'乐团指挥'——一个人指挥 6 个乐手协同演奏。"

### 概念 3：CI/CD = "代码从你的电脑到线上服务器的自动流水线"

很多学员以为部署就是 `scp` + `ssh`。让他们看看没有 CI/CD 的痛苦：

```
你改了一行代码，想上线：

1. git push（推代码）
2. 打开终端，ssh 到服务器
3. git pull（拉代码）
4. pip install（装依赖，如果有的话）
5. 重启服务（kill + 重新运行）
6. 发现报错了，debug...
7. 改好了，重复步骤 1-6

每次上线 10 分钟，一天部署 5 次 = 50 分钟白费
```

**有 CI/CD 之后**：

```
你改了一行代码，git push → 自动测试 → 自动构建 → 自动部署

全程你什么都不用做，3 分钟后自动上线了。
出问题了？自动回滚。
```

**三个概念拆开讲**：

| 概念 | 全称 | 含义 |
|------|------|------|
| **CI** | Continuous Integration | 每次提交代码自动跑测试，保证不引入 bug |
| **CD (Delivery)** | Continuous Delivery | 代码通过测试后自动构建镜像 |
| **CD (Deployment)** | Continuous Deployment | 镜像构建完自动部署到生产 |

> **讲师金句**："CI/CD 不是技术炫技，是**省钱**。你手动部署一次 10 分钟，一年 500 次 = 83 小时。按时薪 100 元算，CI/CD 帮你省了 8300 元/年。"

### 概念 4：反向代理 + HTTPS = "给系统装一个门卫"

学员最困惑的是：为什么要多一个 Nginx？直接暴露 FastAPI 的 8000 端口不行吗？

**不行，原因有三**：

| 问题 | 直接暴露 FastAPI | 加 Nginx |
|------|-----------------|---------|
| HTTPS | 需要在 Python 里处理 SSL | **Nginx 终止 SSL，后端不用管** |
| 限流 | 每个接口自己写 | **Nginx 一行配置搞定** |
| 静态资源 | FastAPI 处理静态文件性能差 | **Nginx 静态文件性能碾压** |
| 多服务 | 只能暴露一个端口 | **一个 80/443 端口代理多个后端** |
| 安全 | FastAPI 直接暴露给公网 | **Nginx 做过滤，隐藏后端** |

> **讲师金句**："Nginx 是系统的门卫——来客先登记（限流），身份不明不让进（HTTPS），快递放门口（静态资源），重要人物请进内室（API 转发）。"

## 三、5 个 Demo，层层递进

### Demo 1：5 分钟跑通第一个容器（15 分钟）

**目标**：让学员从 0 到 1 体验 Docker

```bash
# Step 1: 验证 Docker 安装
docker --version
docker run hello-world
# 看到 "Hello from Docker!" 就算成功

# Step 2: 运行一个 Python 容器
docker run --rm python:3.11-slim python -c "print('Hello from container!')"
# 注意: --rm 表示运行完自动删除容器

# Step 3: 运行一个交互式容器
docker run --rm -it python:3.11-slim python
>>> import sys
>>> sys.version
# Python 3.11.x
>>> exit()

# Step 4: 运行一个 Web 服务
docker run --rm -d -p 8888:8000 python:3.11-slim \
  python -m http.server 8000
# 浏览器打开 http://localhost:8888

# Step 5: 停掉容器
docker ps  # 找到容器 ID
docker stop <container-id>
```

**讲师讲解要点**：

1. `docker run` = 拉镜像 + 创建容器 + 启动，一步到位
2. `-p 8888:8000` = 宿主机 8888 映射到容器 8000（宿主机的端口你自己定，容器的端口应用定的）
3. `-it` = 交互模式（`-i` 保持标准输入，`-t` 分配伪终端）
4. `--rm` = 容器停止后自动删除（适合临时测试）
5. `-d` = 后台运行（detached mode）

**常见问题**：

```
Q: "docker run 好慢？"
A: 第一次要拉镜像（~150MB），之后从本地缓存加载，秒级启动。

Q: "端口冲突怎么办？"
A: 换宿主机端口：-p 9999:8000
```

### Demo 2：为项目 6 编写 Dockerfile（30 分钟）

**目标**：手把手写一个生产级 Dockerfile

```dockerfile
# ============================================================
# 实战: 为多 Agent 系统写 Dockerfile
# 跟着讲师一行一行写，不要复制粘贴
# ============================================================

# 第一步: 选择基础镜像
# 为什么要用 slim? 因为完整版 900MB, slim 只有 150MB
FROM python:3.11-slim AS builder

# 第二步: 设置工作目录
WORKDIR /build

# 第三步: 安装编译工具
# 有些 Python 包（如 cryptography）需要 C 编译器
RUN apt-get update && \
    apt-get install -y --no-install-recommends gcc libffi-dev && \
    rm -rf /var/lib/apt/lists/*
# 注意: rm -rf 清理 apt 缓存，否则镜像会大 50-100MB

# 第四步: 先只复制 requirements.txt
# 这一步利用了 Docker 的层缓存:
# 如果 requirements.txt 没变，Docker 不会重新安装依赖
COPY requirements.txt .

# 第五步: 安装 Python 依赖
RUN pip install --prefix=/install --no-cache-dir -r requirements.txt

# ========================================
# 到这里，builder 阶段完成了
# 下面开始 runtime 阶段（最终镜像）
# ========================================

FROM python:3.11-slim AS runtime
WORKDIR /app

# 第六步: 只装运行时依赖（不装 gcc 等编译工具）
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl tini && \
    rm -rf /var/lib/apt/lists/*

# 第七步: 从 builder 复制已安装的 Python 包
COPY --from=builder /install /usr/local

# 第八步: 创建非 root 用户
RUN useradd -m -u 1000 appuser

# 第九步: 复制应用代码
COPY --chown=appuser:appuser app/ ./app/

# 第十步: 设置环境变量
ENV TZ=Asia/Shanghai \
    PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1

# 第十一步: 切换用户 + 入口点
USER appuser
ENTRYPOINT ["tini", "--"]
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**构建并测试**：

```bash
# 构建
docker build -t multi-agent:v1.0 .

# 查看大小
docker images multi-agent
# 目标: < 500MB

# 运行
docker run --rm -p 8000:8000 multi-agent:v1.0

# 测试
curl http://localhost:8000/health

# 查看镜像层
docker history multi-agent:v1.0
```

**讲师要强调的优化技巧**：

```
优化前:
  FROM python:3.11         → 基础镜像 900MB
  COPY . .                 → 每次改代码都重新安装依赖
  apt-get install ...      → 不清理缓存
  以 root 运行              → 安全隐患

优化后:
  FROM python:3.11-slim     → 基础镜像 150MB（省 750MB）
  COPY requirements.txt .   → 先复制依赖文件
  COPY . .                  → 依赖没变就用缓存
  rm -rf /var/lib/apt/lists → 清理缓存
  USER appuser              → 非 root 运行

最终效果:
  优化前: ~1.2GB
  优化后: ~450MB（省了 750MB = 63%）
```

### Demo 3：Docker Compose 一键启动全套服务（25 分钟）

**目标**：用 Compose 编排 Agent + Redis + Postgres + Chroma

```bash
# 创建项目结构
mkdir -p /tmp/demo-project && cd /tmp/demo-project

# 创建最小化的 docker-compose.yml
cat > docker-compose.yml << 'EOF'
services:
  agent:
    build: .
    ports:
      - "8000:8000"
    environment:
      - REDIS_URL=redis://redis:6379/0
      - DATABASE_URL=postgresql://agent:pass@postgres:5432/agentdb
    depends_on:
      redis:
        condition: service_healthy
      postgres:
        condition: service_healthy

  redis:
    image: redis:7-alpine
    volumes:
      - redis-data:/data
    command: redis-server --appendonly yes
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      retries: 3

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: agentdb
      POSTGRES_USER: agent
      POSTGRES_PASSWORD: pass
    volumes:
      - postgres-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U agent"]
      interval: 5s
      retries: 3

volumes:
  redis-data:
  postgres-data:
EOF

# 启动
docker compose up -d

# 查看状态（每个服务应该是 healthy/running）
docker compose ps

# 查看日志
docker compose logs -f agent

# 进入 Redis 容器验证
docker compose exec redis redis-cli ping
# 应该返回 PONG

# 进入 Postgres 验证
docker compose exec postgres psql -U agent -c "SELECT version();"

# 测试数据持久化
docker compose exec redis redis-cli SET test_key "hello"
docker compose down
docker compose up -d
docker compose exec redis redis-cli GET test_key
# 应该返回 "hello"（数据没丢！）

# 停掉所有服务
docker compose down

# 停掉所有服务并删除数据卷
docker compose down -v
```

**讲师讲解要点**：

1. **depends_on + condition**：确保 Redis 和 Postgres 先启动且健康，Agent 才启动
2. **volumes**：数据持久化的关键。没有 volume，`docker compose down` 后数据全没
3. **healthcheck**：告诉 Docker 怎么判断服务"活着"
4. **容器间通信**：用服务名（如 `redis://redis:6379`），不用 IP 地址

**讲师金句**：

> "Volume 是容器的'外挂硬盘'——容器删了，硬盘还在。**没有 Volume 的数据库容器就是定时炸弹。**"

### Demo 4：GitHub Actions CI/CD 流水线（20 分钟）

**目标**：让学员看到 push 代码后自动部署的完整流程

在课程中可以简化演示（不需要真的部署到服务器，展示配置即可）：

```yaml
# 最小化的 CI 配置（让学员理解流程）
name: Simple CI
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      # Step 1: 拉代码
      - uses: actions/checkout@v4

      # Step 2: 装 Python
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      # Step 3: 装依赖
      - run: pip install -r requirements.txt
      - run: pip install pytest

      # Step 4: 跑测试
      - run: pytest tests/ -v
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: 登录 Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: 构建并推送镜像
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: myuser/multi-agent:latest
```

**讲师讲解要点**：

1. **GitHub Secrets** 是关键——API Key、密码绝对不能写在 YAML 里
2. **`needs: test`** 表示 test 通过后才运行 build（依赖关系）
3. **`on: push`** 表示 push 到 main 分支时自动触发
4. **Actions 生态**——`docker/login-action`、`docker/build-push-action` 都是现成的

```
演示流程:
1. 在 GitHub 仓库里创建 .github/workflows/ci.yml
2. 配置 Secrets（现场演示一遍）
3. git push 到 develop 分支 → 观察 Actions 自动运行
4. 创建 PR 到 main → 观察 CI 运行
5. 合并 PR → 观察 CD 部署
```

### Demo 5：Nginx 反向代理 + HTTPS（20 分钟）

**目标**：让学员理解为什么需要 Nginx，以及 HTTPS 怎么配

先演示"没有 Nginx"的问题：

```bash
# 没有 Nginx: 直接暴露 FastAPI 8000 端口
# 问题 1: 浏览器访问 http://xxx:8000 —— 不专业
# 问题 2: 没有 HTTPS —— 数据裸奔
# 问题 3: 没有 404/500 错误页 —— FastAPI 直接抛 JSON
# 问题 4: 没有限流 —— 一个恶意请求能打挂服务
```

然后写 Nginx 配置：

```nginx
# 最小化的 Nginx 配置（先不加 HTTPS，让学员理解代理概念）
events {
    worker_connections 1024;
}

http {
    upstream agent {
        server agent:8000;
    }

    server {
        listen 80;
        server_name localhost;

        # 反向代理: 用户的请求 → Nginx → Agent
        location / {
            proxy_pass http://agent;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
    }
}
```

```bash
# 测试 Nginx 配置
docker compose exec nginx nginx -t
# 应该输出: syntax is ok, test is successful

# 重载配置（不断服务）
docker compose exec nginx nginx -s reload
```

**HTTPS 讲解要点**：

```
HTTP vs HTTPS:
  HTTP:  数据明文传输，任何人中间截获都能看到内容
  HTTPS: 数据加密传输，即使截获也看不懂

Let's Encrypt:
  - 免费的 SSL 证书颁发机构
  - 证书有效期 90 天（需要自动续期）
  - 申请条件: 你要控制这个域名 + 80 端口可访问

申请步骤:
  1. 买个域名（~55 元/年）
  2. 域名 A 记录指向服务器 IP
  3. 服务器上运行 certbot
  4. 自动获取证书
  5. Nginx 配置证书路径
```

> **讲师金句**："2026 年了，还用 HTTP 部署 API 的，等于把家门钥匙挂在门口。浏览器都会弹'不安全'警告。"

## 四、调试技巧

### 4.1 容器调试三板斧

```bash
# 第一斧: 看日志（90% 的问题靠日志定位）
docker compose logs -f agent           # 实时跟踪
docker compose logs --tail=100 agent  # 最近 100 行
docker compose logs --since 30m agent # 最近 30 分钟

# 第二斧: 进入容器
docker compose exec agent bash         # 交互式 shell
docker compose exec agent python -c "import app; print('ok')"  # 测试导入

# 第三斧: 检查状态
docker compose ps          # 所有服务状态
```

## 五、6 个踩坑点（学员 100% 会遇到）

### 坑 1：端口冲突

**现象**：

```
docker compose up -d
Error: bind: address already in use
```

**原因**：宿主机的 8000 端口已经被其他程序占用了（可能是之前启动的 Python 服务、另一个 Docker 容器等）。

**排查**：

```bash
# 查看谁占用了 8000 端口
lsof -i :8000
# 或者
ss -tlnp | grep 8000
# macOS
sudo lsof -i :8000

# 杀掉占用端口的进程
kill -9 <PID>
```

**预防**：在 docker-compose.yml 中用环境变量控制端口：

```yaml
ports:
  - "${APP_PORT:-8000}:8000"
```

> **讲师金句**："端口冲突是新手最常踩的坑，没有之一。**80% 的学员在 Day 1 都会遇到。** 记住：宿主机端口可以随便改，容器端口必须和应用配置一致。"

### 坑 2：数据丢失

**现象**：`docker compose down` 后再 `docker compose up`，Redis 里的数据全没了。Postgres 的表也消失了。

**原因**：没有配置 Volume。容器的文件系统是临时的，容器删除后文件也删除。

**解决**：

```yaml
# 错误写法（数据在容器内，删除就没了）
redis:
  image: redis:7-alpine
  # 没有 volumes

# 正确写法（数据在 Volume 里，容器删了还在）
redis:
  image: redis:7-alpine
  volumes:
    - redis-data:/data  # 命名卷

volumes:
  redis-data:  # 在底部声明
```

**验证数据持久化**：

```bash
# 写入数据
docker compose exec redis redis-cli SET mykey "hello"

# 删除容器（保留 Volume）
docker compose down

# 重新启动
docker compose up -d

# 验证数据还在
docker compose exec redis redis-cli GET mykey
# 应该返回 "hello"
```

> **讲师金句**："没有 Volume 的数据库 = 把数据库建在内存里。**重启即丢失，上线即翻车。**"

### 坑 3：镜像太大

**现象**：`docker images` 显示镜像 2GB+，推送到 Docker Hub 要等 10 分钟。

**常见原因**：

| 原因 | 多余大小 | 解决办法 |
|------|----------|----------|
| 用 `python:3.11` 而非 `slim` | +700MB | 改用 `python:3.11-slim` |
| 不清理 apt 缓存 | +50-100MB | 加 `rm -rf /var/lib/apt/lists/*` |
| 没有 .dockerignore | +200-500MB | 配置 .dockerignore |
| 不用多阶段构建 | +300MB | 分 builder + runtime |
| COPY 整个项目（含 .git） | +100MB | 配置 .dockerignore |
| 安装了开发工具（pytest、black） | +50MB | 生产镜像不装开发依赖 |

**镜像大小优化目标**：

```
初级: < 1GB（能用）
中级: < 500MB（合格）
高级: < 300MB（优秀）
```

> **讲师金句**："2GB 的镜像推到国内服务器，拉取要 10 分钟。**用户等 10 分钟服务才起来，体验不如不上线。**"

### 坑 4：时区问题

**现象**：日志里显示的时间是 UTC，和中国时间差 8 小时。定时任务全部跑错时间。

**原因**：Docker 容器默认时区是 UTC。

**解决**（在 Dockerfile 中）：

```dockerfile
# 方法 1: 设置 TZ 环境变量（推荐）
ENV TZ=Asia/Shanghai
RUN ln -snf /usr/share/zoneinfo/$TZ /etc/localtime && echo $TZ > /etc/timezone

# 方法 2: docker-compose.yml 中设置
environment:
  - TZ=Asia/Shanghai

# 验证
docker compose exec agent date
# 应该显示 CST（中国标准时间）
```

> **讲师金句**："时区问题特别阴险——**平时不影响功能，一到定时任务就翻车**。日志时间差 8 小时，排问题时你会疯。"

### 坑 5：depends_on 不等于"服务就绪"

**现象**：Agent 启动报错"Connection refused: Redis"。

**原因**：`depends_on` 只保证 Redis 容器启动了，不保证 Redis 服务准备好接受连接。

```
depends_on 的行为:
  Redis 容器启动（进程启动）
  → Agent 容器启动
  → Agent 尝试连 Redis
  → Redis 还在初始化中（端口还没监听）
  → Connection refused!
```

**解决**：用 `condition: service_healthy`：

```yaml
agent:
  depends_on:
    redis:
      condition: service_healthy  # 等 Redis 健康检查通过
    postgres:
      condition: service_healthy

redis:
  healthcheck:
    test: ["CMD", "redis-cli", "ping"]
    interval: 5s
    timeout: 3s
    retries: 3
```

**更安全的做法（应用层重试）**：

```python
import time
from redis import Redis

def connect_redis_with_retry(max_retries=5):
    for i in range(max_retries):
        try:
            r = Redis.from_url("redis://redis:6379/0")
            r.ping()
            return r
        except Exception:
            print(f"Redis not ready, retry {i+1}/{max_retries}")
            time.sleep(2 ** i)
    raise Exception("Cannot connect to Redis")
```

### 坑 6：日志把容器磁盘写满

**现象**：云服务器突然磁盘 100%，所有服务都崩了。

**原因**：Docker 默认日志不限制大小，运行一周后日志占满磁盘。

**解决**（在 docker-compose.yml 中限制日志大小）：

```yaml
agent:
  logging:
    driver: json-file
    options:
      max-size: "50m"   # 单个日志文件最大 50MB
      max-file: "5"      # 最多保留 5 个文件
# 最多占用: 50MB x 5 = 250MB
```

**全局配置**（/etc/docker/daemon.json）：

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "50m",
    "max-file": "5"
  }
}
```

> **讲师金句**："日志不限制 = 磁盘炸弹。**线上运行 3 天，50GB 磁盘被日志吃光，服务器直接趴窝。** 这种坑，踩过一次就够了——不，最好别踩。"

## 六、Q&A 预案

**Q1：为什么不用 Kubernetes？**
A：K8s 对这个项目来说太重了。你需要管理 Pod、Service、Ingress、ConfigMap、Secret、PV/PVC... 学习成本是 Docker Compose 的 10 倍。**先跑通 Compose，等服务真的需要多机部署、自动伸缩时再上 K8s。** 当你有 10 台以上服务器、需要跨机房部署时，K8s 才是必选项。

**Q2：Docker Hub 限速怎么办？**
A：Docker Hub 对免费用户限速（6 小时 200 次拉取）。两个方案：
- 用阿里云 ACR（国内拉取不限速，配置方式见架构建议书）
- 用 GitHub Container Registry（GHCR，免费不限速）

**Q3：为什么用 tini 而不是直接用 CMD？**
A：Python 不会正确处理 SIGTERM 信号（Docker stop 时发的）。tini 作为 PID 1 能正确转发信号，确保优雅停机。**不用 tini = docker stop 可能要等 10 秒超时才停掉。**

**Q4：生产环境要不要用 Alpine 基础镜像？**
A：看情况。Alpine 体积更小（~50MB vs ~150MB），但用 musl libc 而非 glibc，部分 Python 包（特别是 numpy、pandas 的 C 扩展）可能有兼容问题。**用 slim 更安全**，除非你确认所有依赖都兼容 Alpine。

**Q5：Docker Compose 和 docker-compose（带横杠）有什么区别？**
A：`docker-compose` 是 Python 写的 v1 版本（已废弃）。`docker compose`（空格）是 Go 写的 v2 插件。现在都用 v2：`docker compose up -d`。

**Q6：CI/CD 太慢怎么办？**
A：优化三个点：1) Docker 层缓存（cache-from/cache-to），2) 依赖缓存（pip cache），3) 并行任务（test 和 lint 并行）。架构建议书里的配置已经做了这三点优化。

**Q7：一台 2C4G 服务器够吗？**
A：刚起步够用。Redis ~50MB、Postgres ~100MB、Chroma ~200MB、Agent ~500MB、Nginx ~20MB，总共不到 1GB。剩余 3GB 给系统和其他进程。但如果 Agent 处理大量并发，可能需要 4C8G。**先用 2C4G 跑起来，不够再加。**

**Q8：备案太麻烦，不想备案怎么办？**
A：用香港或海外服务器（如阿里云香港区域）可以免备案，但延迟会高 50-100ms。对于 Agent 系统（本身响应就要几秒），50ms 延迟可以接受。成本会贵一些（约 200-300 元/月）。
